# ggen v26.5.19 Perfected Plan

## Authoritative Compiler Wire-Up Release

## 1. Release thesis

**ggen v26.5.19 is the first release where the authoritative path is real.**

That means:

* capabilities resolve to real packs
* packs materialize to disk
* lock state becomes authoritative
* `sync` loads installed pack contributions
* semantic derivation is CONSTRUCT-native
* strict profiles enforce trust/policy
* receipts prove what happened
* CLI surfaces reflect real governed domains
* no success path is stubbed

If any of those are missing, this is not 26.5.4. It is still pre-authoritative.

---

## 2. The single question for every task

Before any work item, ask:

> **Does this make the authoritative compiler path more real?**

If no, it is not v26.5.19 work.

This protects the release from scope drift.

---

## 3. Golden path definition

This is the one flow that must work end to end:

```bash
ggen capability enable --surface mcp --projection rust --runtime axum
ggen pack add mcp-rust
ggen sync --profile enterprise-strict --locked
ggen receipt verify --receipt-file .ggen/receipts/latest.json
```

Success means:

1. capability intent resolves to an actual pack graph
2. pack graph installs and writes authoritative lock state
3. sync consumes installed packs and emits artifacts
4. receipt is written and verifies
5. strict profile is enforced during execution
6. help/output match reality

This golden path should become the **release gate**.

---

## 4. Cut line: must-ship vs not-this-release

### Must ship in 26.5.4

* `registry` noun
* `doctor` noun
* fully wired `pack` commands
* lockfile population and enforcement
* pack loading in `sync`
* profile enforcement in `sync`
* receipt verification path
* fail-closed strict mode
* version normalization for shippable crates
* one end-to-end demo
* updated docs/help

### Not this release

* generalized marketplace UX
* broad economic marketplace features
* adaptive quality-gate tuning
* distributed sync
* federated query execution
* autonomous multi-agent orchestration
* "smart" implicit pack composition without explicit intent

That cut is essential.

---

## 5. Correct product language

### Use at CLI surface

* `registry`
* `pack`
* `capability`
* `policy`
* `receipt`
* `doctor`
* `sync`

### Use conceptually, not necessarily at CLI surface

* marketplace
* economic dimensions
* future n-dimensional selection surfaces

This matters because **registry** is the governed surface, while **marketplace** is the ecosystem concept.

---

## 6. Refined implementation phases

### Phase 0 — Freeze the target

**Goal:** Stop architectural drift before changing code.

1. Freeze the golden path.
2. Freeze the noun-verb CLI surface for 26.5.4.
3. Freeze the must-ship crate list.
4. Freeze the receipt minimum schema.
5. Freeze the lockfile minimum schema.

**Exit criteria:**
* one page release contract exists
* all contributors can answer "what is the 26.5.4 golden path?"
* no open naming dispute about `registry` vs `marketplace`

---

### Phase A — Restore CLI truth

**Goal:** Make the CLI represent real surfaces only.

#### A1. Add `registry` noun

* `registry search <query>`
* `registry info <pack-id>`
* `registry list`

Wire directly to existing marketplace search/metadata infrastructure.

**Exit criteria:**
* `ggen registry search mcp` returns real results
* results come from real registry infrastructure
* help output is accurate

#### A2. Add `doctor` noun

* `ggen doctor`
* `ggen doctor --check <name>`
* `ggen doctor --verbose`
* `ggen doctor --env`

Checks: toolchain, workspace integrity, lockfile integrity, installed pack integrity, trust/profile mismatches, runtime ambiguity.

**Exit criteria:**
* `ggen doctor` runs cleanly in healthy workspace
* unhealthy workspace yields explicit failures/warnings
* no mocked checks

#### A3. Make every `pack` verb real

Required verbs: `add`, `list`, `show`, `remove`, `verify`, `graph`, `update`

**Exit criteria:**
* none of these return placeholders
* each hits real installed state or real registry state
* `pack add` materially installs
* `pack verify` checks signature/digest
* `pack graph` shows real dependency graph

---

### Phase B — Make install state authoritative

**Goal:** Installed pack state must become real and replayable.

#### B1. Normalize materialization path

Define one canonical installed-pack path. No ambiguity. No dual state.

**Exit criteria:**
* all installed packs live under one deterministic location
* path is what sync uses
* doctor validates it

#### B2. Finalize lockfile schema

Minimum fields: identity, version, digest, registry source, trust tier, resolved dependencies, installed path or content-addressed reference.

**Exit criteria:**
* `pack add` writes lock state
* `--locked` execution reads it
* doctor validates consistency
* missing/invalid lock entry fails in strict mode

#### B3. Deterministic bundle expansion

**Exit criteria:**
* `pack add mcp-rust --dry-run` prints exact atomic expansion
* output order is deterministic
* expansion matches installed result

---

### Phase C — Make sync authoritative

**Goal:** `sync` must consume real install state and emit proof-bearing outputs.

#### C1. Load installed pack graph in mu_0

**Exit criteria:**
* installed pack contributions are visible inside sync
* receipt shows which packs participated
* removing a required pack causes sync failure

#### C2. Enforce CONSTRUCT-native derivation

**Exit criteria:**
* SELECT-only or legacy non-authoritative paths are either clearly marked legacy/non-authoritative, or blocked from strict profiles
* templates render from derived graph state
* template-owned semantics are rejected

#### C3. Validation before emission

Minimum checks: required bindings present, no template-owned defaults, no inferred capabilities, no unresolved overlaps, no ambiguous runtime in strict profiles.

**Exit criteria:**
* each failure mode halts before emission
* failure reason is explicit
* no silent degradation

---

### Phase D — Receipts and proof

**Goal:** Every authoritative operation becomes explainable and replayable.

#### D1. Finalize receipt minimum contract

Receipt must contain: pack graph, versions/digests, registry sources, profile, query identifiers, template identifiers, artifact hashes, time/version metadata, optional quality evidence links.

**Exit criteria:**
* receipt written for every successful authoritative sync
* no authoritative sync without receipt

#### D2. Make receipt commands real release gates

Required verbs: `receipt verify`, `receipt info`, `receipt chain-verify`

**Exit criteria:**
* all operate on real receipts
* broken/tampered receipt fails verification
* chain verification detects lineage break

---

### Phase E — Trust, policy, and fail-closed execution

**Goal:** Strict profiles must actually govern execution.

#### E1. Profile enforcement in sync

`ggen sync --profile enterprise-strict` must evaluate: allowed trust tiers, allowed registry classes, required receipt rules, required lock state, runtime constraints.

**Exit criteria:**
* strict profile rejects forbidden pack sources
* strict profile requires locked state when configured
* strict profile rejects unsigned/invalid content

#### E2. Fail-closed rules

Hard fail on: missing truth, trust failure, policy violation, runtime ambiguity, unresolved ownership overlap, broken lock state.

**Exit criteria:**
* all tested explicitly
* no best-effort generation in strict mode

---

### Phase F — Versioning and publication

**Goal:** Ship cleanly without version confusion.

#### F1. Version strategy

* all user-facing/publishable crates: `26.5.4`
* internal path-only crates may remain internal, but only if truly unpublished and invisible
* root: `26.5.4`

#### F2. crates.io dry run

**Exit criteria:**
* all publishable crates pass `cargo publish --dry-run`
* dependency order documented
* README/license/description/repository metadata complete

---

### Phase G — Release proof

**Goal:** Make the release demonstrable and defensible.

#### G1. End-to-end example

Create one canonical example under `examples/v26.5.19-1-0-demo/`.

Must prove: capability resolution, pack install, lockfile population, sync, receipt, receipt verify, strict profile behavior.

#### G2. Release notes

What changed, what is now authoritative, migration from 26.5.4, pack author guidance, profile guidance.

#### G3. Help audit

**Exit criteria:**
* no command claims behavior that is stubbed
* no ghost nouns
* no placeholder examples

## 9. The three most dangerous false positives

1. **CLI completion without authority** — Commands exist, but do not alter the authoritative path.
2. **Sync completion without proof** — Artifacts emit, but receipt path is incomplete or non-authoritative.
3. **Marketplace completion without installation reality** — Registry/search works, but install/lock/sync are not actually connected.

## 10. The minimal demo script that defines the release

```bash
# 1. discover
ggen registry search mcp

# 2. inspect environment
ggen doctor

# 3. install real implementation surface
ggen pack add mcp-rust

# 4. verify installed state
ggen pack list
cat .ggen/packs.lock

# 5. enable intent surface
ggen capability enable --surface mcp --projection rust --runtime axum

# 6. authoritative compile
ggen sync --profile enterprise-strict --locked

# 7. prove it
ggen receipt verify --receipt-file .ggen/receipts/latest.json
ggen receipt info --receipt-file .ggen/receipts/latest.json
```

If this script works cleanly, the release is real.
If this script does not work, the release is not done.

**ggen v26.5.19 is the release where the compiler stops pretending.**
It restores CLI truth, install truth, sync truth, proof truth, and governance truth into one path. It does not ship until a capability intent resolves to real packs, those packs install and lock deterministically, `sync` consumes them authoritatively, receipts prove the result, and strict profiles fail closed on missing truth.
