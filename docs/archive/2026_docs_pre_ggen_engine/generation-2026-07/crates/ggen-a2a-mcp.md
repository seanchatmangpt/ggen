# ggen-a2a-mcp — Generation Audit

**Purpose:** A2A protocol and MCP server for agent-to-agent communication (Cargo.toml). Hosts the
generated A2A protocol types (`a2a_generated/`), a hand-written A2A runtime layer (`a2a/`,
`a2a_registry/`), and the MCP tool surface (`mcp_server.rs`, `mcp_packs.rs`).

**This crate is the workspace's existence proof for the generation-first vision:
67.6% of its code LOC (5,184 / 7,671) is GENERATED today.** It is the only crate in the
workspace containing genuinely generated code.

## Per-module rationale

### `a2a_generated/` — GENERATED (5,184 LOC)

Header of `src/a2a_generated/mod.rs` (lines 1–2), verified:

```
// Generated by ggen from A2A ontology
// Do not edit - regenerate with: ggen sync
```

Contains protocol type surfaces: `adapter.rs` (400), `agent.rs` (92), `converged/agent.rs`
(1,541), `converged/message.rs` (752), `converged/mod.rs` (68), `error.rs` (336),
`handlers.rs` (643), `message.rs` (324), `mod.rs` (378), `port.rs` (435), `task.rs` (215).

**G-ledger caveat (per METHODOLOGY anti-gaming rule):** the header cites `ggen sync`, but
`grep -c 'a2a_generated' ggen.toml` → 0 — no `[[rules]]` entry in the current root manifest
targets this directory, so byte-identity regeneration is **not currently wired/verifiable**.
Templates referencing the module exist (`templates/mcp-server/tool_handler.rs.tera`,
`templates/mcp-server/stdio_server.rs.tera`). Classified GENERATED consistent with the
METHODOLOGY baseline (which counts exactly these 5,184 LOC), but promoting it to the G ledger
requires re-wiring the rule and passing `ggen sync --check`.

### `a2a/` — hand-written runtime layer (1,418 LOC)

- `mod.rs` (136), `artifact.rs` (304), `receipt.rs` (185): serde type definitions for tasks,
  artifacts, receipts — boilerplate-shaped → GENERATABLE-WITH-SPEC (nearest family: the
  a2a_generated type templates themselves).
- `state_machine.rs` (227): `TaskState` transition table + guards. Transition matrices are
  declaratively expressible in TTL (genesis-schema-v2 already models workflow patterns in RDF)
  → GENERATABLE-WITH-SPEC.
- `ggen_construct.rs` (248): MCP tool adapter for `ggen.construct` — JSON schema + dispatch
  wiring per an explicit tool spec (PLAN_INTEGRATION.md §2.2) → GENERATABLE-WITH-SPEC
  (tool-registry pattern; templates/mcp-server/ is the nearest template family).
- `transport.rs` (318): tokio `mpsc`/`RwLock` in-process message-passing between agents —
  concurrency plumbing with channel lifecycle management, no declarative shape →
  IRREDUCIBLY-CUSTOM.

### `a2a_registry/` (531 LOC)

- `types.rs` (52), `error.rs` (30), `query.rs` (53), `store.rs` (78, trait + in-memory impl),
  `registry.rs` (110, thin orchestration over store + monitor), `mod.rs` (13):
  registry/CRUD-shaped → GENERATABLE-WITH-SPEC.
- `health.rs` (182): background tokio health-check loop (interval/timeout/Notify shutdown,
  HTTP pings) → IRREDUCIBLY-CUSTOM.

### MCP surface

- `mcp_packs.rs` (290): wire-protocol layer over `ggen_core::agent::PackAgent`; pure
  `*_result` functions + schemars-typed inputs, shared by MCP and A2A. Tool-table-shaped →
  GENERATABLE-WITH-SPEC (the logic lives in ggen-core; this file is dispatch).
- `mcp_server.rs` (244): rmcp `#[tool_router]` registration wiring → GENERATABLE-WITH-SPEC
  (taxonomy explicitly names CLI/tool wiring; `templates/mcp-server/stdio_server.rs.tera`
  exists but is not wired to this file — arguably GENERATABLE-NOW once a rule + spec are
  confirmed to match; conservatively WITH-SPEC).
- `lib.rs` (17): module re-exports → GENERATABLE-WITH-SPEC.

## Notable

The remaining custom core (transport.rs + health.rs, 500 LOC) is exactly the kind of async
runtime plumbing the E-ledger discussion should adjudicate; everything else in the crate is
either generated today or type/dispatch boilerplate the μ-pipeline can already express.
