
# 30. JAKARTA VALIDATION CONSTRAINTS *


*The constraint lives in the type, not in the if-statement.*


---

## Context

Input validation scattered across controllers and services is inconsistent. Some fields are validated in the controller, others in the service, others not at all. The validation rules for a field (maximum length, allowed pattern) are scattered across multiple files and often duplicated with slight differences. Gaps where invalid data enters the system are common.\n\nSHACL shapes in the ontology are the natural place to declare field constraints: sh:maxLength, sh:pattern, sh:minInclusive, sh:in for enum values. These are formal assertions about the shape of valid data. ggen maps SHACL shapes to Jakarta Validation annotations, generating a single source of truth for all validation rules.

## The Problem

The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.

## Therefore

Express validation constraints in the ontology via SHACL shapes. ggen generates @NotNull, @NotBlank, @Size, @Pattern, @Min, @Max, and custom @Constraint annotations from SHACL property shapes. ## Implementation in ggen sync


ggen reads SHACL NodeShape files and maps each sh:PropertyShape constraint to the corresponding Jakarta Validation annotation. The annotations are generated onto the command records (co-located with the service) rather than onto the entity, keeping the validation at the input boundary rather than deep in the domain.
