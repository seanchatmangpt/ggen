
# 20. TRANSACTIONAL UNIT OF WORK **


*Either everything changes, or nothing changes.*


---

## Context

Operations that span multiple aggregate updates must be atomic. A transfer between accounts must debit one and credit the other in a single transaction. An order placement must create the order, decrement inventory, and emit an event — all or nothing. Partial updates that succeed only halfway leave the system in an inconsistent state that is often discovered long after the transaction has committed.\n\nSpring's @Transactional annotation provides this boundary declaratively, but misuse is common: missing annotations, wrong propagation types, or read-write transactions for read-only queries that prevent database optimization. The unit-of-work pattern makes these boundaries explicit in the ontology.

## The Problem

**Partial updates that succeed only halfway leave the system in an inconsistent state.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Use @Transactional on service methods with explicit READ/WRITE separation. Read-only transactions enable performance optimizations in the database driver and connection pool. The unit-of-work boundary is explicit in the ontology via java26:transactionType — either READ_ONLY or READ_WRITE. ggen generates the correct @Transactional(readOnly=true/false) annotation based on the ontology declaration.\n\nFor multi-aggregate operations, the ontology declares java26:transactionScope with the list of aggregate roots involved. ggen generates a single @Transactional service method that coordinates all aggregate operations within one database transaction.
