
# 33. SCOPED VALUE REQUEST CONTEXT *


*The request carries its context invisibly, accessible anywhere in the call stack without passing parameters.*


---

## Context

Request context (user ID, correlation ID, tenant ID) needs to be accessible throughout the call stack without threading it through every method parameter. The traditional solution is ThreadLocal, but ThreadLocal fails with virtual threads: a virtual thread can be mounted on different platform threads during its lifetime, and ThreadLocal is scoped to the platform thread. Child threads don't inherit parent ThreadLocals.\n\nScopedValue (JEP 446, finalized in Java 26) was designed specifically for virtual threads. A ScopedValue is bound at the entry point of a request and is accessible in the same virtual thread's call stack — including all frames called transitively. Child threads created by StructuredTaskScope inherit the parent's ScopedValues. No parameter passing required.

## The Problem

**ThreadLocal variables fail with virtual threads; passing context explicitly is verbose.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Use ScopedValue (JEP 446) to bind request context at the boundary and access it anywhere in the virtual thread's call stack. Generate the ScopedValue declarations from the ontology. In the ontology, declare java26:RequestContext with the list of context fields (userId, correlationId, tenantId). ggen generates a RequestContext record and a ScopedValues holder class. The Spring filter binds the context via ScopedValue.where().run() at the request entry point.\n\nScopedValues are immutable within a scope: you cannot set a ScopedValue after binding. This eliminates a class of ThreadLocal bugs where code modifies the shared thread-local state unexpectedly.
