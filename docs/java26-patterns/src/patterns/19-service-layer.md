
# 19. SERVICE LAYER **


*Business logic belongs in the service; the controller only routes.*


---

## Context

Spring Boot applications often put business logic in controllers or repositories, creating tight coupling. Controllers grow to hundreds of lines that mix HTTP concerns with domain logic. Repositories become query landfills. The service layer is the logical boundary that most projects neglect until the pain becomes unbearable.\n\nWhen business logic lives in controllers, it cannot be tested without an HTTP server. When it lives in repositories, it cannot be reused without dragging in JPA. The service layer is the only layer that can be tested with real objects, no mocks, and no framework overhead — just plain Java.

---

❖ ❖ ❖

## The Problem

**Business logic scattered across layers is untestable in isolation and unmaintainable.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Generate @Service @Transactional classes that encapsulate all business logic. The service is the only layer that modifies aggregate state. Controllers delegate to services; services delegate to repositories. Command records (Java 26 records) live as inner types of the service they belong to — no separate command package required.\n\nIn the ontology, mark java26:AggregateRoot classes as the service target. ggen generates one service class per aggregate root, with create/read/update/delete operations and the corresponding command records. The service boundary is explicit in the ontology: java26:hasServiceOperation links the aggregate to its allowed operations.

## Implementation in ggen sync


ggen reads java26:AggregateRoot classes and generates a @Service class for each. The generation template (templates/Service.java.tera) produces @Transactional(readOnly=true) on find methods and @Transactional on mutating methods. Command records are generated as inner static records of the service class, co-located with the logic that interprets them.
