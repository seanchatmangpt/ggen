
# 28. PAGINATED QUERY *


*No query returns everything; every list has a page.*


---

## Context

REST APIs that return unbounded lists exhaust memory and time out. findAll() without pagination returns entire tables as the data grows. A table with 10,000 rows today has 1,000,000 rows in three years. The service that worked fine in development fails in production under load, and the failure mode is a timeout or OutOfMemoryError rather than a clean business error.\n\nSpring Data's Pageable interface and Page<T> return type are the standard solution, but they require consistent use across repository, service, and controller. Missing pagination in any layer defeats the purpose.

## The Problem

**findAll() without pagination returns entire tables as the data grows.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Generate paginated repository methods using Spring Data's Pageable + Page<T>. The ontology marks which entities are pageable via java26:pageable true. ggen generates: the repository interface with Page<Entity> findAll(Pageable pageable), the service method that accepts PageRequest, a PaginatedResponse record that wraps Page<T> into a serializable DTO, and the controller endpoint that accepts page, size, and sort query parameters.\n\nA sensible default page size (20) and maximum page size (100) are derived from java26:defaultPageSize and java26:maxPageSize on the entity.

## Implementation in ggen sync


ggen detects java26:pageable true on java26:AggregateRoot and injects Pageable into the generated repository, service, and controller. The PaginatedResponse record includes content, totalElements, totalPages, page, and size — a stable contract that doesn't expose Spring's Page<T> to the HTTP layer.
