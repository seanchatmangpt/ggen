
# 15. VIRTUAL THREAD DAO **


*A thread per request costs nothing when threads are virtual.*


---

## Context

Java 21 introduced virtual threads (Project Loom), making thread-per-request viable for I/O-bound workloads. But JDBC connections and datasources need explicit virtual thread support.\n\nTraditional thread pool DAOs block platform threads while waiting for JDBC results. With virtual threads, the JVM parks the virtual thread during I/O and reuses the underlying platform thread. A system with 10,000 concurrent requests needs 10,000 virtual threads — not 10,000 platform threads — and each JDBC wait parks rather than blocks.

---

❖ ❖ ❖

## The Problem

**Traditional thread pool DAOs waste resources waiting for JDBC; connection pool exhaustion causes cascading failures.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Generate DAO classes that use Executors.newVirtualThreadPerTaskExecutor() for JDBC operations, letting the JVM park virtual threads efficiently during I/O waits.\n\nEach generated DAO wraps JDBC calls in virtual thread tasks submitted to a dedicated per-task executor. Results are collected via Future<T>.get(). The virtual thread parks during the JDBC wait and is rescheduled when data arrives.

## Implementation in ggen sync


The virtual-thread-dao template generates a DAO class per java26:AggregateRoot that has java26:dataAccess java26:VirtualThreadDAO. Each read method submits a Callable<T> to a static Executors.newVirtualThreadPerTaskExecutor() and calls Future.get(). Write methods follow the same pattern. The executor is created fresh per operation to avoid reuse-after-close bugs.
