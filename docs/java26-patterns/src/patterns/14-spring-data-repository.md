
# 14. SPRING DATA REPOSITORY **


*A query is a question about the domain; the repository answers it without revealing the implementation.*


---

## Context

Data access in Spring Boot uses Spring Data JPA repositories. Writing these by hand for 20 entities means 20 nearly identical interfaces with custom query methods scattered throughout.\n\nThe naming conventions for derived query methods (findBySkuAndActiveTrue, existsByEmail, countByStatus) are powerful but easy to misspell. A method named findBySkuAndActvieTrue (typo in Active) compiles but fails at startup. Generating these names from the ontology eliminates the typo class entirely.

## The Problem

**Hand-written repositories duplicate the pattern and diverge from the entity model.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Define the entity as java26:AggregateRoot in the ontology; ggen generates a typed JpaRepository<Entity, UUID> with the entity's natural queries (findByField, existsBy, etc.).\n\nFields declared java26:queryBy true become derived finder methods. Fields declared java26:unique true generate existsBy and findBy variants. The repository interface name follows the pattern ${ClassName}Repository.

## Implementation in ggen sync


The spring-data-repository template generates a Spring Data JPA repository interface. It extends JpaRepository<Entity, UUID> and adds derived query methods for fields marked java26:queryBy true. The interface is annotated @Repository and placed in the same package as the entity.
