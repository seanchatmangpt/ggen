
# 31. SPRING CACHE ABSTRACTION *


*Read many times, compute once — the cache is the memory of the service.*


---

## Context

Read-heavy services benefit from caching but cache configuration is scattered across the codebase. @Cacheable annotations are added inconsistently. Some findById methods are cached but not findAll. Cache eviction is forgotten after save and delete, leaving stale entries. The cache key strategy is inconsistent across services, making cache warming and invalidation scripts brittle.\n\nWhen caching rules are derived from the ontology, they are applied consistently. The same generator that adds @Cacheable to findById also adds @CacheEvict to save and delete. The cache name is derived from the entity class name, making it predictable.

---

❖ ❖ ❖

## The Problem

**Manually added @Cacheable annotations diverge from the actual access patterns.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Mark read-heavy entities as java26:cacheable in the ontology. ggen generates @Cacheable, @CacheEvict, and @CachePut on the appropriate service methods. @Cacheable(\

## Implementation in ggen sync


ggen detects java26:cacheable true on java26:AggregateRoot and generates the @EnableCaching configuration, the @Cacheable/@CacheEvict/@CachePut annotations on service methods, and a CacheConfig inner class that defines the cache TTL from java26:cacheTtlSeconds.
