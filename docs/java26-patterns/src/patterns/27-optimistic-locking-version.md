
# 27. OPTIMISTIC LOCKING VERSION *


*The version field is the entity's fingerprint — it changes every time the entity changes.*


---

## Context

Concurrent updates to the same entity can silently overwrite each other without locking. Two transactions read an entity at version 1. Both modify it. Both save it. The second save overwrites the first without any error — a lost update. This class of bug is invisible in unit tests and only appears in production under concurrent load.\n\nJPA's @Version annotation implements optimistic locking: on UPDATE, JPA adds WHERE version = N to the SQL. If another transaction has incremented the version since we read it, the update affects zero rows and JPA throws ObjectOptimisticLockingFailureException. The service layer can catch this and retry or surface a conflict error to the client.
