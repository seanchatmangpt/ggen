
# 13. JPA ENTITY MAPPING **


*A table in the database and a class in Java are the same fact, expressed twice.*


---

## Context

JPA entity mapping requires @Entity, @Table, @Id, @Column, @GeneratedValue, and often @Version, @CreationTimestamp, etc. Writing these by hand is error-prone and inconsistent.\n\nEach entity must mirror the database schema, carry JPA lifecycle annotations, enforce nullability constraints, and participate in the optimistic locking protocol via @Version. Doing this by hand for 30 entities means 30 opportunities to misspell a column name, forget a constraint, or omit the protected no-arg constructor that JPA requires.

---

❖ ❖ ❖

## The Problem

**Manually written JPA entities diverge from the actual database schema over time.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Define each entity once in the ontology as java26:AggregateRoot with its fields. ggen generates the @Entity class with correct annotations, a JPA repository, and a Flyway migration in one sync.\n\nThe ontology is the single source of truth: field names, types, nullability, and uniqueness constraints all live in the TTL. ggen reads them once and emits consistent annotations across every generated artifact.
