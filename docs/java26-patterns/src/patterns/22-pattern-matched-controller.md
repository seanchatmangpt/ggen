
# 22. PATTERN-MATCHED CONTROLLER **


*When the request type determines the handler, the switch statement is the architecture.*


---

## Context

REST APIs often have endpoints that dispatch to different handlers based on request type or state. A payment endpoint might accept card payments, bank transfers, or voucher redemptions. A command bus pattern accepts different command types. Traditional if-else chains for dispatch are error-prone: adding a new variant requires remembering to update the dispatch logic in a separate file.\n\nJava 26 sealed interfaces and pattern matching switch eliminate this class of bug. The sealed hierarchy of request variants is defined in the ontology; the switch expression guarantees exhaustive handling at compile time. If a new variant is added to the ontology, the generated switch will fail to compile until all variants are handled.

## The Problem

**if-else chains for dispatch are error-prone and don't enforce exhaustive handling.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Model the request variants as a sealed interface in the ontology. Generate a @RestController that uses pattern matching switch to dispatch, ensuring all variants are handled. The sealed hierarchy in the ontology maps directly to a generated sealed interface hierarchy in Java 26, and the switch expression is generated from the variants' handling rules.\n\nIn the ontology, use java26:PatternMatchedAPI with java26:hasVariant to enumerate all request types. Each variant maps to a handler method in the generated service.
