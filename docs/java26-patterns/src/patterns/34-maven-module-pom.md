
# 34. MAVEN MODULE POM *


*The build file is just another projection of the domain's dependency graph.*


---

## Context

Multi-module Maven projects require consistent pom.xml files with correct parent references, dependency management, and plugin configuration. Manually maintained pom.xml files diverge from each other: one module uses Java 21, another Java 26. One module uses Spring Boot 3.2, another 3.4. The Surefire plugin version is inconsistent. These divergences cause build failures that are hard to trace and time-consuming to fix.\n\nThe dependency relationships between modules are a graph: module B depends on module A. This graph is part of the domain's information architecture and belongs in the ontology. Once the module graph is in the ontology, generating consistent pom.xml files is straightforward: ggen reads the module declarations and generates a parent pom and all module poms from the same template.

## The Problem

**Manually maintained pom.xml files diverge from each other and from the dependency graph.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Generate pom.xml files from the ontology's module declarations. The parent pom and all module poms are consistent because they come from the same source. In the ontology, declare java26:MavenModule instances with their groupId, artifactId, Java version, and dependencies. ggen generates the parent pom with <dependencyManagement> and all module poms with the correct <parent> reference and <dependencies>. ggen sync regenerates pom.xml whenever the ontology changes.\n\nA BOM (Bill of Materials) pom is also generated from java26:BomModule — centralizing dependency version management across all modules.

## Implementation in ggen sync


ggen reads java26:MavenModule declarations and generates one pom.xml per module in the correct directory. The parent pom aggregates all modules in <modules>. Dependency versions are centralized in the parent <dependencyManagement>. Plugin versions are standardized across all modules from a shared plugin configuration in the parent.
