
# 17. FLYWAY SCHEMA MIGRATION **


*The migration script is the ontology's changelog, made executable.*


---

## Context

Every change to the JPA entity model requires a corresponding Flyway migration. Writing these by hand creates opportunities for schema drift.\n\nThe typical workflow: developer adds a field to the Java entity, forgets to write the migration, pushes to CI, and the application fails at startup because Flyway detects the schema is out of sync. Or the migration is written but uses the wrong column name. The ontology knows the current state of the model; generating migrations from it closes the loop.

---

❖ ❖ ❖

## The Problem

**Manually written Flyway migrations diverge from the entity model, causing runtime failures.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Generate the Flyway migration SQL directly from the ontology. When a field is added to the ontology, ggen generates both the updated @Entity and the V{N}__add_{field}.sql migration.\n\nThe migration template uses the same field metadata (name, type, nullability) as the entity template, guaranteeing consistency. The version number N is derived from the current count of migration files in db/migration/.

## Implementation in ggen sync


The flyway-migration template detects newly added java26:Field nodes (those with no prior migration file) and generates an ALTER TABLE ADD COLUMN migration. The SQL type is mapped from the java26:fieldType using a lookup table in the template: String → VARCHAR(255), BigDecimal → NUMERIC(19,4), UUID → UUID, int → INTEGER, Long → BIGINT. The file is written to src/main/resources/db/migration/.
