
# 21. REST CONTROLLER **


*The controller is a thin adapter between HTTP and the domain.*


---

## Context

REST controllers often grow fat with business logic, validation, error handling, and transformation code. The controller becomes the dumping ground for everything that doesn't clearly belong elsewhere. Teams add direct repository calls, conditional business logic, and manual JSON transformation — all in the @RestController class.\n\nA fat controller cannot be tested without an HTTP server. It cannot be reused in a CLI context. It couples the HTTP transport to the domain logic. When the HTTP API changes shape (REST to GraphQL, REST to gRPC), the entire fat controller must be rewritten.

---

❖ ❖ ❖

## The Problem

**Fat controllers that contain business logic cannot be tested without an HTTP server.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Generate @RestController classes that only: validate input via @Valid, delegate to @Service, transform entities to DTOs via static factory methods, and map exceptions to HTTP status codes via @ExceptionHandler. The controller never touches a repository. It never contains an if-statement for business logic. It is a pure HTTP adapter.\n\nIn the ontology, java26:AggregateRoot with java26:exposeAsRest true triggers REST controller generation. The endpoints are derived from java26:hasServiceOperation — each operation becomes one HTTP endpoint.

## Implementation in ggen sync


ggen generates one @RestController per java26:AggregateRoot that has java26:exposeAsRest true. The template (templates/RestController.java.tera) derives HTTP verbs from operation names: find* → GET, create* → POST, update* → PUT, delete* → DELETE. A companion @ControllerAdvice is generated for exception-to-HTTP-status mapping.
