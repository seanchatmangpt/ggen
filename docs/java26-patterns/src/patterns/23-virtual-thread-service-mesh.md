
# 23. VIRTUAL THREAD SERVICE MESH **


*A thousand calls, a thousand virtual threads, one thread pool that never fills.*


---

## Context

Microservice architectures make frequent outbound HTTP calls. Traditional thread pools become bottlenecks: a slow downstream service holds a platform thread captive, and the pool fills up. Under load, the service stops accepting new requests while threads wait on I/O that hasn't completed yet. Thread pool exhaustion causes cascading failures throughout the service mesh.\n\nJava 26 virtual threads (JEP 444, finalized) eliminate this problem. A virtual thread blocked on I/O releases the carrier thread back to the pool. The JVM can manage millions of virtual threads with the same handful of platform threads. The application code looks like blocking code but behaves like non-blocking code.

## The Problem

**Thread pool exhaustion under load causes cascading failures in service meshes.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Configure all HTTP clients to use virtual threads. Generate service mesh client classes that use structured concurrency (JEP 453) to make multiple parallel calls safely. In the ontology, declare java26:RemoteService dependencies with java26:callStrategy — either SEQUENTIAL, PARALLEL, or FIRST_SUCCESS. ggen generates the correct StructuredTaskScope usage for each strategy.\n\nThe application server configuration is also generated: spring.threads.virtual.enabled=true in application.properties is derived from java26:useVirtualThreads true on the application's ontology root.

## Implementation in ggen sync


The service mesh template generates one client class per java26:RemoteService. For PARALLEL strategy, the client uses StructuredTaskScope.ShutdownOnFailure with multiple fork() calls. For FIRST_SUCCESS, it uses StructuredTaskScope.ShutdownOnSuccess. For SEQUENTIAL, it generates plain blocking calls (virtual threads make blocking fast enough).
