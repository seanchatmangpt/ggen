
# 16. DTO PROJECTION **


*The API speaks a different language than the database; the DTO is the interpreter.*


---

## Context

JPA entities are mutable and persistence-aware. Exposing them via REST APIs leaks persistence concerns and causes over-fetching.\n\nReturning a @Entity directly from a REST controller serializes lazy-loaded collections (causing N+1 queries), exposes internal state mutations, and couples the API version to the database schema. Every schema change breaks the API contract. DTOs decouple these two planes of evolution.

---

❖ ❖ ❖

## The Problem

**Returning JPA entities from REST controllers couples the API contract to the database schema.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Use java26:ValueObject in the ontology to generate immutable DTO records with a from(Entity) factory method and toEntity() method. Records' structural equality and immutability make them safe to serialize.\n\nThe DTO record carries only the fields that the API surface needs, not every persistence field. The from() factory extracts them from the entity. The toEntity() method reconstructs a new entity for writes, enforcing the invariants at construction time.

## Implementation in ggen sync


The dto-projection template generates a Java record for each java26:ValueObject that has java26:projectsFrom pointing to a java26:AggregateRoot. The record components mirror the selected fields. A static from(Entity e) factory method is generated using field access. A toEntity() method is generated for entities that support it.
