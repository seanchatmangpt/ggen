
# 24. STRUCTURED CONCURRENCY SCOPE **


*A scope that opens, lives, and closes — no orphaned threads, no forgotten futures.*


---

## Context

Asynchronous operations often leave threads orphaned or resources leaked when errors occur. CompletableFuture chains have no lifetime: if the thread that started them is interrupted, the downstream futures may continue running indefinitely, consuming resources and producing results nobody is waiting for. Cancellation is manual and error-prone. Exception propagation through chained futures is confusing.\n\nJEP 453 (Structured Concurrency, finalized in Java 26) solves this by giving concurrent operations a lexical lifetime. A StructuredTaskScope is opened like a try-with-resources block. All forked tasks are guaranteed to complete before the scope closes. If any task fails, the scope can cancel the rest. No threads escape the scope.

## The Problem

**Unstructured concurrency (CompletableFuture chains) makes error propagation and cancellation unpredictable.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Use StructuredTaskScope (JEP 453) to scope concurrent operations. All forked threads complete within the scope's lifetime. Generate scope-based task implementations from java26:TaskScope in the ontology. The ontology declares the scope strategy (SHUTDOWN_ON_FAILURE or SHUTDOWN_ON_SUCCESS) and the set of concurrent tasks. ggen generates the try-with-resources scope block with all forks, the join, and the result assembly.\n\nThis pattern is composable: scopes can be nested. The outer scope owns the lifetime of all inner scopes.

## Implementation in ggen sync


ggen reads java26:TaskScope definitions and generates a static method or inner class per scope. The ShutdownOnFailure strategy is used when all results are required (batch fan-out). The ShutdownOnSuccess strategy is used when the first successful result wins (redundant service calls, hedging). Both are generated from the java26:scopeStrategy property.
