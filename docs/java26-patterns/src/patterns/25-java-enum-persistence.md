
# 25. JAVA ENUM PERSISTENCE *


*The string 'PENDING' in the database is the fact; the enum in Java is the interpretation.*


---

## Context

Domain status fields need persistence as readable strings, not ordinals. Storing enums as ordinals (0, 1, 2) is fragile: reordering enum variants in source code changes the meaning of every row in the database without changing the code. String persistence avoids this, but requires consistent naming conventions and a factory method for deserialization.\n\nThe ontology is the ideal place to declare enum variants, since they are facts about the domain. The valid states of an OrderStatus are part of the domain model, not implementation details. Expressing them in OWL ensures they are available for SHACL validation, documentation generation, and Java code generation from a single source.

## The Problem

**Storing enums as ordinals breaks when variants are reordered; storing as strings requires consistent naming.**


The forces at play create tension between competing needs. Without resolution, these forces lead to suboptimal outcomes—complexity where simplicity is needed, fragility where robustness is required, opacity where clarity is essential.


---

## Therefore

Generate Java enums with @Enumerated(EnumType.STRING), a getValue() method returning the canonical string, and a fromValue(String) factory. Use the ontology to define valid variants via java26:Enum with java26:hasVariant. Each variant's rdfs:label becomes both the Java name and the database string value. The fromValue() factory throws an IllegalArgumentException with a descriptive message listing the valid values, making debugging easy.

## Implementation in ggen sync


ggen reads java26:Enum class definitions and generates one Java enum per class. The template includes the @Enumerated(EnumType.STRING) annotation, a private final String value field, a constructor, getValue(), and fromValue() factory. Variants are generated from java26:hasVariant in declaration order.
