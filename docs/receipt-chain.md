# Cryptographic Receipt Chain

## Overview

ggen's receipt chain is an **Ed25519 hash-linked log** that provides tamper-evident
proof of provenance for every generated artifact. Each receipt cryptographically binds
a file's content hash to the ontology version, toolchain version, and commit hash that
produced it.

This document explains how the chain works, what it proves, and how to validate it.

## Architecture

```
Ontology (TTL) ──┐
                 ├── ggen sync ──> Generated Code ──> Receipt (Ed25519 signed)
Template (Tera) ─┘                                      │
                                                         │
                                                         v
                                              .ggen/receipts/chain.json
                                                         │
                                                         v
                                              Hash-linked chain:
                                              receipt[N-1] ──hash──> receipt[N]
```

### What Each Receipt Contains

| Field | Purpose |
|-------|---------|
| `file_path` | Relative path of the generated file |
| `file_hash` | SHA-256 of the file content at generation time |
| `file_size` | Byte size for quick sanity checks |
| `commit_hash` | Git commit that triggered generation |
| `generated_at` | ISO 8601 timestamp |
| `generator` | Tool that created the receipt |
| `generator_version` | Version of the receipt generator |

### Chain Structure

The chain is a JSON array stored at `.ggen/receipts/chain.json`. Each entry
includes a `previous_chain_hash` field linking it to the hash of the entire
chain as it existed before the current entry was appended. This makes the
chain **append-only** and **tamper-evident**: modifying any historical receipt
breaks the hash link of every subsequent entry.

## What It Proves

A valid receipt chain proves:

1. **Provenance**: This file was generated by ggen (not hand-written or copied)
2. **Integrity**: The file content has not been modified since generation
3. **Lineage**: The file was produced from a specific ontology + template version
4. **Temporal ordering**: Receipts are ordered by generation time
5. **Non-repudiation**: Ed25519 signature binds the receipt to the signing key

### What It Does NOT Prove

- The ontology itself is correct (that requires separate validation)
- The generated code compiles or passes tests (that requires CI)
- The template logic is sound (that requires code review)

## Validation

### Interactive Validation

```bash
# Validate all receipts for generated files
ggen sync --validate-only

# Validate with verbose output
ggen sync --validate-only --verbose

# Validate a specific file's receipt
ggen receipt verify --file path/to/generated.rs
```

### Pre-commit Gate

The receipt gate runs automatically before each commit:

```bash
# Install the gate (adds to .git/hooks/)
ggen hook install --receipt-gate

# Or manually:
ln -sf ../../scripts/ggen-receipt-gate.sh .git/hooks/pre-commit-receipt
```

The gate checks:
- All generated files in the staging area have corresponding receipts
- Stored file hashes match current file content
- The receipt chain is structurally valid

**Strict mode** (exits 1 if any receipt is missing):
```bash
ggen-receipt-gate.sh --strict
```

### Post-commit Generation

Receipts are generated automatically after each commit:

```bash
# Install post-commit hook
ln -sf ../../scripts/ggen-receipt-generate.sh .git/hooks/post-commit-receipt
```

### Manual Receipt Generation

```bash
# Generate receipts for current HEAD
ggen-receipt-generate.sh

# Generate receipts for a specific commit
ggen-receipt-generate.sh abc1234
```

## CI Integration

### GitHub Actions

```yaml
name: Receipt Validation
on: [pull_request]

jobs:
  validate-receipts:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Need full history for chain validation

      - name: Install ggen
        run: cargo install ggen-cli --locked

      - name: Validate receipt chain
        run: |
          ggen sync --validate-only --strict
          echo "Receipt chain valid"

      - name: Check for tampered files
        run: |
          ./scripts/ggen-receipt-gate.sh --strict
```

### Makefile Integration

```makefile
.PHONY: receipts-validate receipts-generate

receipts-validate:
	@./scripts/ggen-receipt-gate.sh --strict

receipts-generate:
	@./scripts/ggen-receipt-generate.sh
```

## Security Properties

### Tamper Detection

If a generated file is modified without regenerating its receipt, the pre-commit
gate detects the mismatch between the stored SHA-256 hash and the current file hash.
The gate exits 1, blocking the commit.

### Chain Integrity

The chain uses hash-linking: each entry contains `previous_chain_hash`, which is the
SHA-256 of the entire chain JSON as it existed before the entry was appended. Any
modification to a historical entry invalidates all subsequent hash links.

### Key Management

- Signing keys are stored in `.ggen/receipts/signing_key.json`
- Add `signing_key.json` to `.gitignore` (keys should NOT be committed)
- For team use, rotate keys periodically with `ggen receipt rotate`
- Public keys can be committed for verification without signing capability

### Threat Model

| Threat | Mitigation |
|--------|------------|
| File tampering | SHA-256 hash comparison in pre-commit gate |
| Receipt tampering | Chain hash-linking detects modification |
| Key compromise | Key rotation; `.gitignore` prevents key commits |
| Chain rollback | Hash-linking; CI validates against main branch chain |
| Replay attacks | Commit hash binding; timestamps |

## Troubleshooting

### "MISSING receipt" warnings

Generated files exist without receipts. Fix:
```bash
ggen sync --audit
```

### "TAMPERED receipt" errors

File content has changed since receipt was generated. Fix:
```bash
# If changes are legitimate, regenerate:
ggen sync --audit

# If tampering is suspected, investigate:
git diff HEAD -- path/to/file
git log --oneline -5 path/to/file
```

### "Malformed chain file"

The chain JSON is corrupt. Fix:
```bash
# Backup and regenerate
cp .ggen/receipts/chain.json .ggen/receipts/chain.json.bak
ggen-receipt-generate.sh
```
