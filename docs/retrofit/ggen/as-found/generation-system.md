## 4. Sync pipeline stages (`crates/ggen-engine/src/sync.rs`, 2578 lines) — as read, not run

Module doc (lines 1-19) states five stages: Resolve -> Enrich -> Extract -> Render -> Write,
plus a receipt-chain step. Reading `pub fn sync` (starts line 152) against the actual code
shows the real stage boundaries (span names, not just the doc comment's prose) are:

1. **Stage 0 — Schema dispatch** (line 169): `crate::schema_dispatch::load(root)` classifies
   `ggen.toml` as either `DeclarativeRules` (`[[generation.rules]]`, routed to
   `crate::generation_rules::run`) or `Frontmatter` (this function's own `GgenConfig`). One
   shared dispatch point — confirmed by the doc comment's claim that `handle_doctor`/
   `handle_graph_validate`/`build_law_engine` call the same function, not a copy.
2. **Stage 1 — Resolve / `pipeline.load` span** (line ~180): reads `ggen.toml`'s
   `[ontology].source` TTL, inserts it into a `GraphLawStore` (default) or `DeterministicGraph`
   (oxigraph) engine, resolves and unions packs' `ontology.ttl` + declared
   `extra_ontology_paths`, discovers templates, and binds an explicit input closure
   (`BTreeMap<String,String>` of relative-path -> BLAKE3-or-"MISSING") covering the actuator
   version, project name, `ggen.toml` itself, the ontology, every pack ontology/extra-ontology,
   and every template file plus its `from:`/`rdf:` frontmatter-declared inputs. A declared
   input that is unreadable at bind time is recorded as the literal string `"MISSING"` in the
   closure rather than silently dropped — an explicit as-found/as-desired gap marker, not a
   silent narrowing.
3. **P2 reflexive receipts** (still inside Stage 1, gated on `config.law.reflexive`): parses
   this project's own prior `.ggen-v2/receipt-log.jsonl` and inserts one `ggenr:Sync` triple
   cluster per successfully-parsed line before any template renders, so later SPARQL can see
   the project's own sync history as graph facts. This is the one documented exception to
   fail-closed: a malformed prior receipt-log line is recorded into the closure as
   `"receipt-log:MALFORMED-{n}"` and skipped, rather than bricking every future sync — the
   code comment (lines ~345-355) explicitly names this as the only sanctioned silent-skip
   point in the whole pipeline.
4. **Stage 2 — Enrich / `pipeline.extract` span** (line ~379): runs each template's
   `construct:` SPARQL CONSTRUCT once against the shared graph and inserts the produced
   triples — explicitly single-pass, not iterated to a fixed point (a construct that depends
   on another construct's output needs a second `sync` run; documented, not silently assumed
   fixed-point). Templates that declare an RDF overlay (`declares_rdf_overlay`) are skipped
   here — their `construct:` instead runs later against a private per-template overlay graph,
   preventing that template's `rdf:`/`rdf_inline:` facts from leaking into every other
   template's shared-graph view.
5. **Stage 2b — Law / `pipeline.validate` span** (line ~409): optional (`[law]` table absent
   -> whole stage no-ops); loads N3/Datalog rule files, materializes, and runs denial/SPARQL
   gates. Engine-conditional: `EngineKind::Oxigraph` typed-refuses (`[FM-LAW-*]`) any
   configured law input rather than silently ignoring it.
6. **Stage 3/4 — Extract+Render / `pipeline.generate` span** (line ~667): per-template `when:`
   ASK-guard evaluation, named `sparql:` SELECT execution, Tera render.
7. **Stage 5 — Write / `pipeline.emit` span** (line ~901): applies `crate::write::plan_write`
   Hygen-style write semantics, then (non-dry-run only) computes and chains the
   `praxis_core::ReceiptRecord` and appends `.ggen-v2/receipt-log.jsonl`.

This 7-checkpoint breakdown (Stage 0 dispatch + P2 reflexive-receipt sub-step are real code
branches the 5-stage doc-comment summary doesn't name) is what the Mermaid diagram renders.

## 5. Write/file-ownership semantics (`crates/ggen-engine/src/write.rs`)

`plan_write`'s freeze-policy precedence, read directly from the module doc comment and
`check_freeze`/`record_freeze_checksum`:

1. No `freeze_policy` set -> normal Create/Overwrite/Inject semantics apply (mode is a
   template-frontmatter field, not detailed further in this pass — Agent A's
   `repository.md` is the file of record for CLAUDE.md's documented mode=Create semantics).
2. `freeze_policy: always` and the target file already exists -> `Skipped`, unconditionally,
   regardless of content diff.
3. `freeze_policy: checksum` -> the target is skipped only if its on-disk BLAKE3 no longer
   matches the checksum ggen itself recorded on a prior successful write (stored under
   `freeze_slots_dir`); a target that still matches what ggen last wrote is not frozen.
   `freeze_slots_dir` is required when `freeze_policy: checksum` is set — its absence is a
   typed `[FM-WRITE-006]` error, not a silent default. `freeze_slots_dir` values are validated
   against path traversal (`../`) and absolute paths (both covered by real unit tests in the
   same file, `freeze_slots_dir_traversal_is_err`/`freeze_slots_dir_absolute_is_err`).
4. After any successful `Written`/`Injected` outcome under `freeze_policy: checksum`, the new
   content's checksum is recorded for the next run's comparison — the freeze state is
   self-maintaining, not hand-edited.
