## Notable: the 2 cells that reached L4

### star-toml-pack — Handler-gap size

Both skeptics agree reaches_l5=false but diverged on current_level (skeptic 1: L4; skeptic 2: L3). Independent verification breaks the tie toward L4. Reading templates/star_toml_config.rs.tmpl directly: StarTomlConfig::load() (lines 46-55) is a one-line delegation to `star_toml::load_file::<Self>(path)`, and the surrounding structs are entirely template-generated from the a_sections/fields SPARQL queries -- there is no seam anywhere for a consumer to plug in domain-unique logic, because none is needed for the module to function. This is confirmed at the process level by crates/ggen-engine/tests/framework_packs_e2e.rs: scaffold_pack_project (lines 40-58) writes only an empty ontology.ttl and a ggen.toml for the consumer project -- never a hand-authored .rs file -- and `ggen sync run` still succeeds, producing a fully-formed config module (the test asserts the structs, deny_unknown_fields, and the load() delegation are all present in the generated text). This is stronger than L3's bar ('only domain-unique logic is hand-written, behind a stable seam' -- which still presumes some hand-written logic exists, just isolated): here there is zero hand-written domain logic, required or optional. It matches L4's bar verbatim: 'Hand-written code is optional extension, never required for the subsystem to work.' Skeptic 2's L3 evidence, on inspection, actually argues why the pack doesn't reach L5 (the deny_unknown_fields/loader choice is hardcoded by the template author rather than specified in the ontology) rather than giving concrete evidence for L3 over L4 -- that argument is about behavioral origin (relevant to Ontology expressiveness and the L5 gap), not about whether a consumer must write code, which is what the Handler-gap dimension measures.

**Still not L5 because:** L5 requires zero handler gap because behavior itself is specified in the ontology or composed from another pack's generated logic -- not because no consumer code happens to be required. Here the admission policy (deny_unknown_fields/closed-world, choice of loader crate) is hardcoded once by the pack author into the Tera template; ontology.ttl has no property asserting 'reject unknown keys' or naming a loader engine. Concrete gap: add ontology-level facts (e.g. a SHACL closed-shape/cardinality constraint or an explicit stp:validation property) that the template merely projects rather than hardcodes, or demonstrate composition with another pack's generated validator, before the already-zero consumer-side handler gap could also be credited as ontology-specified behavior.

### self-monitoring-pack — Fire precision

Beyond L2's fire/no-fire fixture pair (pattern-fires.ttl / pattern-does-not-fire.ttl, both passing) and L3's topic-boundary-confusion case (pattern-different-topic.ttl, proven not to fire), direct inspection of self_monitoring_real_session_actuation.rs confirms two genuine L4-shaped adversarial checks with real, passing evidence: (1) adversarial_run_and_blocker_responses_never_fire_inside_real_session_graph constructs synthetic, reserved-topic-literal GroundingQuestion->RunResponse/BlockerResponse pairs (a 'gamed' input engineered specifically to probe the rule's boundary) and injects them directly into the real, dense 10326-triple session graph; both independent SPARQL engines (oxigraph CONSTRUCT and TripleStore materialize()) agree on zero firings. (2) README's live-verified SHACL tamper test substitutes an out-of-scheme smon:NotARealConcept for smon:GroundingQuestion (a 'malformed' input) and confirms admission fails closed -- exit 1, naming both offending focus nodes and the exact violated shape message. Two of L4's three named adversarial categories (malformed, gamed) are directly evidenced by real, executed, passing tests, not merely a topic-confusion pair, which is why L4 (not L3) is the better-supported reading.

**Still not L5 because:** L5 requires precision 'measured on real captured data over time, not only fixtures.' All of this pack's real-data adversarial work is a single one-time retrospective analysis of ONE archived transcript (session-real.ttl), captured and analyzed once, with no formal precision/recall metric computed and no repeated/longitudinal measurement across multiple sessions or a continuously running capture stream -- the two confirmed real false negatives (turn-1656/1663 topic split; turn-1657's missed SurveyResponse classification) are a narrative write-up from that single snapshot, not a tracked score re-measured over time.

## Per-pack gap detail

The specific, concrete missing capability per (pack, dimension) — the actual L2→L5 or L3→L5 roadmap, not a restatement of "more work needed."

### Codegen packs

#### clap-noun-verb-pack

- **Generation depth** (L2): Zero business-logic vocabulary exists in ontology.ttl (cnv:Command carries only noun/verb/handler-name-string/doc/returnType-string/args-string — pure function-signature metadata). Reaching L5 requires the ontology to express what each handler actually does (state changes, validation, computed output) so templates could render handler bodies themselves, plus generate the surrounding crate scaffolding (mod wiring, Cargo.toml deps) — eliminating both the mandatory external handlers.rs file and the hand-authored mount points, so nothing outside `ggen sync`'s output is required to compile and run the subsystem.
- **Handler-gap size** (L2): No RDF property lets a cnv:Command's behavior be expressed in the ontology itself, and no declarative cross-pack composition property exists (e.g. a hypothetical cnv:delegatesTo linking a command directly to another pack's already-generated symbol) that would let the template wire behavior without a human writing a Rust fn. Until one of those two exists, 100% of commands require a hand-written handler; the handler gap is total, not zero, and L5 ('zero handler gap') is out of reach.
- **Ontology expressiveness** (L2): A different template set could regenerate identical function signatures from this ontology, but never an equivalent system, because no behavior is encoded — only shape. Reaching even L3 requires the ontology to state relations/cardinality/error cases so templates become thin projections instead of hardcoding structure; L5 requires encoding what each command does (e.g. structured pre/post-condition or effect triples per cnv:Command) sufficient that swapping templates (Rust to another language) would still produce a functionally equivalent CLI — today a different template author would have to invent 100% of the behavior independently.
- **Consumer effort** (L2): For consumer effort to reach L5 ('Consumer wires ggen.toml. Done.'), the handler-gap and mod-wiring requirements both have to disappear — they are the same underlying gap viewed from the consumer's side. As long as handlers.rs/mod.rs/lib.rs must be hand-authored for the CLI to compile and run, consumer effort cannot exceed L2/L3 regardless of how simple ggen.toml itself is; two concrete artifacts (handlers.rs, the mod declaration) would need to be generated or made unnecessary automatically by `ggen sync` alone.
- **Test generation** (L3): Passing this suite certifies only that the wrapper-to-handler wiring compiles and doesn't panic — it does not certify correct CLI behavior. L5 requires the proof suite alone to be sufficient evidence the subsystem works, which requires: (a) assertions on actual output content rather than just `.is_ok()`, (b) at least one real failure/error path per command, (c) exercising the actual `#[verb]`-macro-driven CLI dispatch rather than bypassing it via direct function calls, and (d) deriving expected literals from the same SPARQL that produces the routes (removing the human-transcription step) so exact-count (`==`) checks become safe against duplication or corruption.
- **Regeneration lifecycle** (L2): L5 requires 'drift is impossible by construction (lock + receipt refuse it)' — today drift in the generated file is only prevented if a developer remembers to add the literal marker comment, and drift in the actual business-logic files is not detected or refused at all since no hash tracking exists for them. Closing this gap requires either bringing handler files into the receipted/hashed closure with refusal-on-mismatch semantics, or eliminating the hand-written lane (per the handler-gap dimension) so there is nothing left outside ggen's tracked outputs to drift, plus an automatic ontology-change to regen-and-diff-with-receipt flow (the L4 bar) wired for this pack specifically.
- **Target-API fidelity** (L2): L5 requires fidelity to be definitional, not checked — i.e. the pack would need to track the target's ontology-level contract rather than its current Rust API, so an API-level rename or signature change in clap-noun-verb couldn't break generation. Today the opposite is true: the ontology and templates are literal transcriptions of clap-noun-verb's current API (macro name, registry function signature, distributed-slice mechanism), so any such change requires hand-editing this pack. Closing the gap requires this pack's ontology.ttl to be mechanically derived from the target crate itself (e.g. from its macro-expansion output or a published machine-readable schema), plus automated detection when the pinned target version changes — not merely a one-time human alignment check with no CI enforcement even at the lower L3 bar.

#### wasm4pm-compat-pack

- **Generation depth** (L2) *(skeptics disagreed; resolved by direct re-check)*: L5 requires the entire crate surface (types, logic, tests, docs) to precipitate from RDF with no human in the loop. - **Handler-gap size** (L2): Ontology would need to specify id/timestamp derivation and firing conditions (which pipeline stage triggers which event) so no hand-written orchestration is needed, and that composed call site would need to be actually wired into ggen-engine's/ggen-graph's real sync pipeline. Concrete work item: change `w4pm:objectType "Pack"` to `w4pm:objectType w4pm:Pack` (a real IRI) and add per-event attribute-property declarations plus cardinality/ordering facts (e.g. graph_union_hashed -> pack_lock_verified -> receipt_chained sequencing), so templates stop hardcoding structure the ontology doesn't state.

- **Target-API fidelity** (L2): Fidelity would need to be definitional, not checked: wire this proof into CI so a target version bump is automatically re-verified (closing the L2->L3 gap first), then pursue an actual ontology-to-ontology tracking relationship (this pack's ontology derived from or linked to wasm4pm-compat's own upstream schema) so a target-crate field rename is detected structurally rather than requiring another hand-verification pass.

- **Consumer effort** (L2): The pack must auto-wire the generated module into the consumer's crate (e.g. generate or patch a mod-registration point, or use an auto-registration pattern) so src/wasm4pm_crate_catalog.rs is reachable without a hand-written `mod wasm4pm_crate_catalog;` somewhere in lib.rs/main.rs. Today that manual step is required, which keeps the pack at 'writes... mod declarations are the consumer's problem' (L2), not 'Consumer wires ggen.toml. Done.' (L5) -- and more fundamentally, the pack delivers a static reference table rather than a working subsystem, so there is no working system for 'done' to certify in the first place.
- **Test generation** (L3): The proof suite only certifies internal self-consistency between the generated catalog and this pack's own hand-transcribed ontology.ttl -- it never checks against the actual live ~/wasm4pm workspace (no fixture ingest of real Cargo.toml files, no drift detection, no CI re-derivation), is not wired into CI or `just pre-commit`, and requires manual hand-update whenever ontology.ttl changes (a human stays in the maintenance loop, the explicit L3-not-L4 marker). There are also no error paths to cover, since the pack generates only static data. L5 ('sufficient evidence on its own... certifies the subsystem') is further blocked because there is no nontrivial subsystem being certified here -- only a flat const array -- so reaching L5 on this dimension is gated on Generation depth and Handler-gap advancing past L2/L1 first.
- **Regeneration lifecycle** (L2): There is no freeze/inject mechanism letting hand-edits and generated content coexist across regens (the L3 bar -- absent here since the pack generates no hand-editable seam at all), no semantic-diff-plus-receipt-on-upstream-change flow (L4), and no lock+receipt mechanism that structurally refuses drift (L5). Concretely, today a change to ontology.ttl silently overwrites all three output files with no diff surfaced to the consumer -- a hand-edit to the generated files would simply be silently clobbered on the next sync rather than refused or flagged.
- **Target-API fidelity** (L1): L5 requires the pack to track the target's ontology rather than scrape its API/manifest at all -- the exact opposite of what this pack does, since it is defined entirely by reading Cargo.toml text. Closing this gap is a conceptual redesign (the target would need to expose an ontology this pack could depend on), not an incremental fix. Short of that, the concrete gap chain is: add a single dated hand-verification note (L2), then wire a CI check that compiles/greps against a pinned wasm4pm checkout (L3), then auto-detect and re-verify on a version bump of that target (L4) -- none of which exist today.

#### mfact-pack

- **Generation depth** (L2): No logic, only static data derived from a 3-stage/5-directory README description. To reach L5 the ontology would need to specify actual executable behavior of the mfact/procint pipeline (not just label/order/actor/path facts about it), and templates would need to emit a compilable module that performs a function, not merely restate facts as Rust consts -- and the test suite (see Test generation) would need to stop being hand-transcribed so that logic, not just data and docs, genuinely precipitates from RDF.
- **Handler-gap size** (L1): The dimension is currently undefined because there is no behavior surface at all. To reach L5 ('behavior itself is specified in the ontology'), the ontology would first need to model actual pipeline behavior (what Project/Admit/Certify concretely DO, not just their order/actor/label/comment metadata), and templates would need to generate that behavior directly -- today there is nothing to close a gap around.
- **Ontology expressiveness** (L2) *(skeptics disagreed; resolved by direct re-check)*: No second, independent template set has ever been built or tested against this ontology to demonstrate an 'equivalent system' could be regenerated, and the ontology encodes only flat, per-class scalar facts (order/actor/path) with no relations between PipelineStage and AuthorityDir, no cardinality constraints, and no behavioral/pre-post-condition contracts -- all of which L3 ('structure: relations, cardinality, error cases') through L5 require before template-independence could even be attempted.
- **Consumer effort** (L2): Ship a template that emits (or appends to) the consumer's lib.rs/main.rs with the required `mod mfact_catalog;`/`pub use` wiring, so nothing beyond editing ggen.toml's [packs] table is required to get a compiling result -- currently the mod declaration is a manual, undocumented step left entirely to the consumer.
- **Test generation** (L3): By explicit design, passing this proof suite does NOT certify the subsystem against current ontology truth -- it only certifies agreement with a frozen, human-maintained snapshot that can silently drift out of sync. To reach L5 the proof would need to re-derive its expected values from the live ontology at test time (or otherwise structurally guarantee sync), and coverage would need to extend past the 8 fixed individuals to error/edge paths, so that a passing run is self-sufficient evidence rather than evidence contingent on a human having remembered to update literals.
- **Regeneration lifecycle** (L2): Regen cannot currently be 'the only maintenance verb' -- the pack itself documents that a human must hand-edit the generated test file after an ontology change for the anti-drift check to stay meaningful. L5 needs a lock+receipt mechanism that detects this exact semantic-diff case and either regenerates the test expectations automatically or refuses the sync, instead of silently leaving a stale, always-identical test file in place.
- **Target-API fidelity** (L1): mfact/procint would need to expose its own canonical machine-readable ontology that this pack directly tracks, eliminating the README-transcription step entirely, before 'fidelity is definitional, not checked' could even be attempted. Currently there is not even an L2-level 'verified by hand against one target version, once (dated note)' -- no such note exists in any of the pack's files.

#### mfw-pack

- **Generation depth** (L2): No compilable module performing any function is generated (no parse/dispatch/emit code) -- only data and docs, and even the test file is explicitly not SPARQL-derived. To approach L5 ('types + logic + tests + docs' precipitating from RDF), the pack needs an actual behavior surface -- e.g. a stage-transition validator or lookup/dispatch keyed by CycleStage.order -- and the proof's expected values must be derived from the same query path as the catalog rather than hand-typed, so logic and tests both precipitate from ontology.ttl instead of only types and docs.
- **Handler-gap size** (L1): Before a handler gap can even be measured, the pack must first generate a real behavioral entry point (see Generation depth). L5 requires 'behavior itself specified in the ontology' with zero hand-written glue; currently the ontology (ontology.ttl) encodes only labels/order/path/comment facts, no behavior at all, so there is no seam to close yet -- the concrete missing capability is any generated function whose body does real work driven by ontology data.
- **Ontology expressiveness** (L1): The ontology models static facts about mfw's directory layout and stage names, not mfw's actual behavior (transition/validation semantics for the cycle, what a receipt must contain). L5 requires the ontology to be 'a complete specification; a different template set could regenerate an equivalent system' -- since no behavior is encoded, no template set could regenerate anything beyond the same static table. The concrete missing capability is encoding the cycle's actual transition/validation semantics as RDF relations, not just an ordering integer.
- **Consumer effort** (L2): L5 bar is 'Consumer wires ggen.toml. Done.' Reaching it requires an inject-mode template that adds the necessary `mod mfw_catalog;`/`pub mod` declaration into the consumer's existing src/lib.rs automatically, so no manual module-wiring step remains after `ggen sync run`. No such injection template exists in packs/mfw-pack/templates/ today.
- **Test generation** (L3): L5 requires the proof suite to be 'sufficient evidence on its own' certifying the subsystem. Here the proof is explicitly not self-sufficient: it cannot detect a real SPARQL/template regression because its expectations are copied by hand from the same source a human already read, not independently re-derived; a human must notice test failures after an ontology change and manually re-transcribe. Closing the gap requires deriving the proof's expected values from a path independent of the catalog template (e.g. a second, differently-shaped SPARQL query or a checked-in fixture snapshot) so passing the suite requires no human transcription step, plus wiring it into CI so 'passing' is continuously demonstrated rather than a one-off scratch run.
- **Regeneration lifecycle** (L2): L5 requires drift to be 'impossible by construction (lock + receipt refuse it).' Nothing here refuses or even flags the exact drift vector this pack creates (proof literals silently diverging from ontology.ttl after edits) -- regen blindly clobbers instead of detecting the mismatch. Closing the gap needs either a self-deriving proof (removing the human-transcription drift vector entirely) or a generation-time check that fails the sync when the catalog's derived row counts/values disagree with checked-in proof literals, rather than deferring detection to a later, separate test run.
- **Target-API fidelity** (L1): L5 requires the pack to track the target's ontology structurally, making fidelity 'definitional, not checked.' Here fidelity is not even checked once against a dated snapshot (the weaker L2 bar) -- it is a hand-authored restatement of AGENTS.md prose with a known-incomplete crate-description gap. Closing the gap requires importing/consuming mfw's actual RDF ontology files (under its own ontology/ surface) directly as this pack's source of truth instead of hand-transcribing AGENTS.md, so structural drift in the real mfw repo is caught by construction rather than requiring a human to reread AGENTS.md and re-author facts.

### Knowledge-hook packs

#### self-monitoring-pack

- **Derivation power** (L2): Needs (a) a second/parameterized obligation class with severity or deadline fields usable by a downstream planner (L3 -- EscalationObligation today only carries two turn-links + a reason string), (b) obligation chaining into ordered discharge plans (L4), and (c) an actual standing process that re-derives and re-prioritizes as new turns are appended live, rather than one static SPARQL CONSTRUCT invoked once per test run (L5).
- **Actuation closure** (L2) *(skeptics disagreed; resolved by direct re-check)*: No actuation step exists at all: no shipped script fires on a derived smon:EscalationObligation (the L3 bar), no receipt is minted when the obligation derives, and no re-observe step feeds any actuation outcome back into the graph (L4). A complete unattended derive->actuate->receipt->re-observe loop with receipted refusal paths (L5) would need to be built from nothing -- today the only 'actuation' is a SPARQL SELECT run inside a Rust #[test] to inspect what materialize() derived.
- **Input acquisition** (L3) *(skeptics disagreed; resolved by direct re-check)*: L4 needs continuous, SHACL-gated ingest of real events, not one manual script invocation producing one static file with no admission check on the actually-captured data. L5 needs the pack itself to own capture + admission + retention end to end so a consumer only points it at a source, rather than a human running a Python script by hand and managing the output file themselves.
- **Fire precision** (L4) *(skeptics disagreed; resolved by direct re-check)*: L5 requires precision 'measured on real captured data over time, not only fixtures.' All of this pack's real-data adversarial work is a single one-time retrospective analysis of ONE archived transcript (session-real.ttl), captured and analyzed once, with no formal precision/recall metric computed and no repeated/longitudinal measurement across multiple sessions or a continuously running capture stream -- the two confirmed real false negatives (turn-1656/1663 topic split; turn-1657's missed SurveyResponse classification) are a narrative write-up from that single snapshot, not a tracked score re-measured over time.
- **Obligation lifecycle** (L2): No discharge/refusal vocabulary or status property exists at all, so there is nothing to query for open vs. closed obligations. Needs a discharge/refusal predicate and state vocabulary (L3), overdue-obligation escalation via the same hook mechanism (L4), and a receipted, replayable full ledger from derivation through discharge/refusal (L5) -- currently there is not even a minimal 'open obligations' query surface that persists past a single test process.
- **Composability** (L2): No proven co-fire test over a union graph of >=2 packs' hooks exists (L3), let alone hooks referencing other packs' derived facts by design (L4) or an RDF-declared precedence scheme across hook families (L5) -- there is currently only one kh:Hook pack in the whole repository, so 'hook families forming a governance stack' has no second family to stack with yet. A second kh:Hook pack plus a real union-graph co-fire test is the concrete next step.
- **Governance coverage** (L2): Only one governed pattern exists out of the whole conversational-turn process (no governance of BlockerResponse follow-through, RunResponse quality, or any pattern besides this one escalation rule), and no meta-mechanism anywhere flags turns/transitions falling outside every governed pattern. L5 requires both broad happy+exception-path coverage of the whole target process and a catch-all 'ungoverned transition' detector that itself positively fires when no known pattern matches.

#### dogfood-lifecycle-pack

- **Derivation power** (L1): No derivation mechanism exists at any level, so a 'standing governor that re-plans as new facts arrive' is unreachable without first building the L2 floor: a hook.ttl with at least one SPARQL CONSTRUCT that derives a parameterized obligation from dfl:ToolEvent facts (e.g. an obligation to review dfl:Blocked events). Only after that exists does re-planning/prioritization become a meaningful next step.
- **Actuation closure** (L2): Needs all four L5 elements, none of which exist today: (1) an actually-wired unattended trigger, not a README instruction to run a command by hand; (2) a real derive->actuate causal link (impossible until Derivation power exists); (3) refusal receipting -- the guard's exit-2 block path must append a receipt the way session-end.sh does for the success path, and the capture hook's dead Blocked branch must become reachable; (4) actuation outcomes re-entering the graph as new observed facts (closed loop), which nothing here does.
- **Input acquisition** (L3): Parameterize the capture/validation target (env var, CLI flag, or config-file setting) instead of a hardcoded absolute path to a sibling repo on one machine; add real on-ingest SHACL validation inside the capture hook itself (not a later manual batch step) to close the L4 gap first; and define an explicit retention/rotation policy for the append-only log, which currently grows unboundedly with no pruning.
- **Fire precision** (L2): Commit the adjacent-case fixture the README currently only narrates (to make the L3 claim reproducible), add adversarial/gamed fixtures for L4, and, for L5, build an actual precision measurement (a computed fire/no-fire accuracy rate tracked over a real, pack-reachable corpus of captured sessions over time) -- none of which exists in any form today; the current 'real data' is one developer's local files in an unrelated repository.
- **Obligation lifecycle** (L1): Requires Derivation power to exist first (an actual derived obligation node), then discharge recording, then overdue escalation, then a full replayable ledger tying derivation to discharge/refusal -- none of the four pieces exist; there is no subject matter for lifecycle tracking yet.
- **Composability** (L2): Author actual RDF predicates (e.g. a dfl:precedes / priority vocabulary term) declaring firing order or precedence relative to at least one other hook pack already in this repo (self-monitoring-pack is the natural candidate), plus a real test proving the two packs' hooks fire in the declared order over one union session graph -- nothing resembling either exists today.
- **Governance coverage** (L1) *(skeptics disagreed; resolved by direct re-check)*: Build a genuine coverage-detection mechanism -- e.g. a session-wide tool-call counter (from Claude Code's own hook-invocation count or transcript) cross-checked against the number of captured dfl:ToolEvent nodes, with any gap surfaced as a flagged/receipted anomaly. Nothing like this exists today: every capture failure or filtered-out tool call is a silent, unrecorded exit 0, and the one blocking mechanism is dormant by default and inoperative in this repo (its `cng` dependency crate does not exist). This is also in tension with the pack's own stated observation-only design philosophy, which would need to be revisited, not merely extended, to claim any transition-level coverage.

- **Question coverage** (L1): Zero committed SPARQL queries exist anywhere in the pack, so even L2 is unmet. To reach L3 the pack needs, per CQ, a committed .rq file (or embedded query in a test) plus a test asserting the expected answer against fixtures/case.ttl; L4 needs coverage across the full CQ set with regression protection. Today the only adjudication logic is the one narrow amount-threshold hook producing a bare derived triple -- no rationale, no receipt, and no path for structurally new cases (L4), which must exist before L5's fully-unattended bar is even attemptable.
- **Domain fidelity** (L2): No domain-expert acceptance process or artifact exists at all -- the vocabulary-fit and threshold work is self-conducted research, not externally reviewed. Before L5 is reachable, the pack needs L3 (clause-by-clause tracing to real source material -- the threshold/waiting-period values are illustrative, not sourced to current statutory text) and L4 (validation against real-world deal data, not fictional fixtures); L5 additionally requires a named domain expert accepting the corpus's verdicts as authoritative, which presupposes both steps plus a verdict mechanism to accept (see Verdict authority).

Before that: the identifier-collision resolution code the pack's own fixture cites as proof does not exist in this repository, and the pack's own disclosed LEI-shape gap means at least one deliberately-gamed input is NOT refused today, so L4 ('deliberately gamed inputs are refused with reasons') is not fully met either.
- **Planning integration** (L2): Needs the referenced grounding/solving pipeline (bcinr-pddl / powl2-decompose, which DO exist as real workspace crates) actually wired to pddl-domain.ttl, a real passing test producing a discharge plan, then an automated verdict-to-plan-to-discharge-to-receipt pipeline. None of this exists; pddl-domain.ttl is currently disconnected RDF data carrying a false 'proven live' citation.
- **Generalization** (L2) *(skeptics disagreed; resolved by direct re-check)*: Only one full-coverage case instance exists against the pack's own schema (ontology.ttl+shapes.ttl); the other fixtures are narrow single-purpose proof fixtures, not additional independent domain cases, and no automated SHACL conformance check has ever run against any of them. Before L5 ('any conforming instance from any source is admissible'), the pack needs at minimum >=2 independent FULL case instances machine-validated unchanged against the schema (L3), then generated/templated instance authoring instead of hand-written Turtle (L4), then an actual external-source admission pipeline (L5) -- none of which exist; every instance to date was hand-authored in the same pass that wrote the schema.
- **Reasoner independence** (L2): No second reasoning engine has ever reproduced any of this pack's derivations (L3's bar is unmet), no exportable SHACL report or plan trace exists for external checking (L4), and no receipt artifact exists that could carry an engine-agnostic proof (a precondition for L5's literal wording: 'the receipt carries the proof'). All three of L3, L4, and L5 are simultaneously unmet, with L3 (a second engine) the nearest concrete next step.

## Method

Two independent read-only agents ("skeptics") audited every pack against its type's exact L5
bar text (quoted verbatim in each agent's prompt from `docs/packs/PACK_MATURITY_MODEL.md`),
instructed to default to `reaches_l5=false` and require a specific, cited file+content match to
the literal L5 wording before scoring anything higher. Skeptics did not run builds/tests
(read-only, to avoid 40-way concurrent `cargo test` contention on shared example projects) —
they read `packs/*/ontology.ttl`, every `templates/*.tmpl`, and already-committed generated
artifacts (e.g. `examples/receiptctl/src/*.rs`, `examples/receiptctl/tests/*.rs`) as evidence.
A third agent per pack then reconciled the two skeptics' verdicts, breaking any disagreement by
re-reading the pack directly and favoring the more skeptical (lower) level unless concrete
evidence supported the higher one — per this repo's own calibration rule that ties resolve
toward not claiming maturity.

Full raw skeptic transcripts and per-agent JSON are in the workflow journal
(`subagents/workflows/wf_fc79a041-695/journal.jsonl` under this session's Claude project
directory) for anyone who wants to audit the auditors.
