# DFLSS for `stpnt` ⚙️🔥

Use **DMADV** (Design for Lean Six Sigma), not DMAIC.

```text
Define → Measure → Analyze → Design → Verify
```

## Define

Start with the canon, not the feature.

```text
Genesis 1:1
→ Scripture Basis Set
→ AA Steps / Traditions / Concepts
→ Avatar Set
→ Gall Checkpoint
→ Stewardship Obligation
```

Definition statement:

> `stpnt` exists to prevent people from being silently lost between invitation and incorporation into the body.

## Measure

Critical-to-quality (CTQ) measures:

| CTQ | Measurement |
| :--- | :--- |
| Welcome | person was received and remembered |
| Consent | no follow-up without consent |
| Ownership | every obligation has a steward |
| Timeliness | follow-up occurs before deadline |
| Closure | every opened path terminates lawfully |
| Continuity | no silent disappearance |
| Multiplication | received person can eventually receive another |

## Analyze

Primary defects:

```text
silent visitor loss
unowned follow-up
consent violation
phantom membership
stale steward assignment
orphan ministry path
false completion
leader bottleneck
```

Core failure mode:

> **CRUD records people but does not steward them.**

## Design

Design around movable stewardship parts:

```text
Event
→ Hook
→ Admit / Refuse
→ Actuate
→ Receipt
→ Next Event
```

v0 parts:

```text
WelcomeOneAnotherPart
ConsentGatePart
AssignStewardPart
FollowUpObligationPart
PentecostInvitationPart
ContinuityWatchPart
```

## Verify

A part is promoted only if it proves:

```text
scripture basis exists
AA structure mapped
owner exists
timer exists
terminal state exists
receipt emitted
no private namespace
no silent loss path
```

## Clean Line

> **DFLSS makes `stpnt` impossible to design from taste. Every feature must prove canon, obligation, owner, timer, terminal state, and receipt.**
