## 2. Level 5 capability model (12 required capabilities)

| # | Capability | Definition |
|---|---|---|
| 01 | Authoritative semantic source | The pack's ontology is the sole source of truth for what gets built; no fact lives only in a template or hand-written module. |
| 02 | Complete generation surface | Every file the consumer needs (types, logic, tests, docs, provenance, release surface) is generated -- no hand-authored production file required. |
| 03 | Deterministic regeneration | The same ontology + templates produce byte-identical output on repeated ggen sync run, verified live, not asserted. |
| 04 | Fixed-point convergence | Regeneration from a fully-deleted consumer converges to the same result as incremental regeneration -- erase-and-rebuild is a real, exercised path. |
| 05 | Generated verification | The proof suite is itself generated from the ontology (not hand-written against generated code), and its expected values derive from ontology facts, not hardcoded literals. |
| 06 | Generated negative witnesses | Sabotage/falsification fixtures (compile-fail, refusal-path, adversarial input) are generated, proving the pack fails closed, not just that it succeeds. |
| 07 | Generated documentation | Reference docs are projected from the ontology, not hand-written prose describing generated code after the fact. |
| 08 | Generated provenance | Every generated artifact traces to the ontology facts and template version that produced it (source-manifest / digest chain), machine-checkable. |
| 09 | Generated receipts | Sync emits a cryptographic receipt covering this pack's contribution, chained and verifiable (ggen receipt verify). |
| 10 | Generated release surface | Packaging/CI/release artifacts for the generated subsystem are themselves generated, not hand-maintained alongside it. |
| 11 | Generated evolution path | An upstream ontology or target-API change flows through to a re-synced consumer with a semantic diff and no manual repair. |
| 12 | Consumer replacement | A consumer with zero knowledge of the target's own implementation can delete its generated tree and regenerate a working, tested, receipted subsystem from ggen.toml + the pack alone. |
