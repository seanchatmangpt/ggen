# Usability & CLI Audit Handoff Report

## 1. Observation

During the exploration of the `ggen` CLI interface, the following direct observations were made:

### A. Mismatch in Parameter Casing and Value Expectation
When running the `ggen init` command with standard POSIX kebab-case flags (e.g., `--skip-hooks`), the parser failed:
```bash
$ target/debug/ggen init --path /Users/sac/ggen/.agents/teamwork_preview_explorer_usability_audit_2/test_init_project --skip-hooks
error: unexpected argument '--skip-hooks' found
  tip: a similar argument exists: '--skip_hooks'
Usage: ggen init --path <PATH> --skip_hooks <SKIP_HOOKS>
```
The auto-generated command help output lists these fields as snake_case value-bearing options, whereas the manually written markdown description in the help text lists them as kebab-case flags:
* **Verbatim Help Section (Markdown doc comment)**:
  ```
  ## Flags
  --path PATH               Project directory (default: current directory)
  --force                   Overwrite existing files
  --skip-hooks              Skip git hooks installation
  ```
* **Verbatim Options Section (Auto-generated by clap)**:
  ```
  Usage: ggen init [OPTIONS]

  Options:
        --path <PATH>                
        --force <FORCE>              
        --skip_hooks <SKIP_HOOKS>    
        --name <NAME>                
        --version <VERSION>          
        --description <DESCRIPTION>  
    -h, --help                       Print help
  ```

### B. Incompatibility of Scaffolder Manifest (`ggen.toml`) with Parser
The manifest `ggen.toml` generated by `ggen init` contains fields and sections that the `ggen sync` command's manifest parser rejects.
* **Observation 1**: Initialized `ggen.toml` contains `ontology.standard_only = true`. When parsing, `ggen sync` fails:
  ```
  ERROR: CLI execution failed: Command execution failed: error[E0001]: Manifest parse error
    --> ggen.toml
    |
  26 | standard_only = true
     | ^^^^^^^^^^^^^
  unknown field `standard_only`, expected one of `source`, `imports`, `base_iri`, `prefixes`
  ```
* **Observation 2**: Initialized `ggen.toml` contains `project.authors = ["ggen init"]` and `project.license = "MIT"`. Parser fails:
  ```
  unknown field `authors`, expected one of `name`, `version`, `description`
  ```
* **Observation 3**: Initialized `ggen.toml` contains sections `[sync]`, `[rdf]`, `[templates]`, and `[output]`. Parser fails:
  ```
  unknown field `output`, expected one of `project`, `ontology`, `inference`, `generation`, `validation`, `packs`
  ```

* **Code Verification**: 
  * In `/Users/sac/ggen/crates/ggen-core/src/manifest/types.rs:50`: `GgenManifest` is decorated with `#[serde(deny_unknown_fields)]`.
  * `ProjectConfig` (line 78) defines only `name`, `version`, and `description`.
  * `OntologyConfig` (line 93) defines only `source`, `imports`, `base_iri`, and `prefixes`.
  * In `/Users/sac/ggen/crates/ggen-cli/src/cmds/init.rs:124`, the hardcoded `ggen.toml` template writes out the unsupported fields:
    ```rust
    standard_only = true
    ```
    and the sections:
    ```rust
    [sync]
    ...
    [output]
    ...
    ```

### D. RDF Syntax Errors in `ontology.ttl`
Running `ggen graph validate` on `/Users/sac/ggen/ontology.ttl` fails due to syntax errors:
```
ERROR: CLI execution failed: Command execution failed: Validation failed: Extraction failed: Failed to load ontology: Failed to load Turtle: Parser error at line 100 between columns 5 and 29: "Load A2A ontology file" is not a valid predicate
```
* **Code Verification** (in `/Users/sac/ggen/ontology.ttl:99`):
  ```turtle
  ggen:steps [
      "Load A2A ontology file",
      "Check existence of 7 domain module classes",
      "Verify inheritance from abstract base classes",
      "Validate property definitions for each class"
    ] ;
  ```
  This blank node does not contain valid predicate-object pairs; instead, it uses a JSON-like list of strings, which is syntactically invalid Turtle.

## 2. Logic Chain

1. **Observation A** shows a parser failure when entering standard kebab-case options like `--skip-hooks`, while the `--help` command lists kebab-case under `## Flags` but snake_case under `Options:`. 
2. The logic chain shows that `clap-noun-verb` macro processes Rust function parameter names (e.g. `skip_hooks: Option<bool>`) directly to CLI parameters, resulting in snake_case options (`--skip_hooks`) that require value arguments (e.g. `--skip_hooks true`), causing documentation/behavior discrepancies and usability friction.
3. **Observation B** and the code files in `ggen-core` and `ggen-cli` confirm that two different configurations are defined: `GgenConfig` in `ggen-config` (which contains fields like `authors`, `license`, `standard_only`, `output`) and `GgenManifest` in `ggen-core` (which does not contain these fields and denies unknown fields).
4. Since `ggen sync` uses `GgenManifest` to parse the `ggen.toml` manifest file, any `ggen.toml` generated by `ggen init` (which follows the template containing fields from `GgenConfig`) will fail parsing, preventing the scaffolded project from being synced out-of-the-box.
5. **Observation C** shows that application-level errors (like duplicate init or missing files) output a JSON response containing `status: "error"` but return exit code `0`. This is because error handling in these subcommands captures the error and returns a successful `VerbResult::Ok` containing the error representation instead of returning a `VerbResult::Err`, preventing shell scripts/CI pipelines from detecting failures via exit codes.
6. **Observation D** shows that the project's root `ontology.ttl` contains a list of strings wrapped in brackets (`[ ... ]`) without predicate-object relations. This is invalid Turtle syntax, which causes the RDF parser to reject the file.

---

## 3. Caveats

* The investigation was performed purely in a read-only manner, and no codebase files were modified. The test project was initialized and modified solely within the agent's dedicated directory `/Users/sac/ggen/.agents/teamwork_preview_explorer_usability_audit_2`.
* Observability stacks (Tempo/Jaeger/OTel) were not running, causing `ggen doctor` to mark them as failed (which did not prevent the main CLI check from passing).
* The actual generation output was only verified using `--dry_run true` since generating files in the workspace root could violate the read-only constraint.

---

## 4. Conclusion

1. **Usability Gaps**: The CLI suffers from argument casing issues (snake_case instead of kebab-case, and value options instead of boolean flags) which contradict the manual help pages.
2. **Scaffolding Breakage**: The `ggen init` command generates a manifest `ggen.toml` that is completely incompatible with the `ggen sync` parser, causing a broken out-of-the-box experience.
3. **Error Handling & Exit Codes**: Process exit codes are not consistently set to non-zero values when errors occur, making automated integration and CI/CD piping fragile.
4. **Data Defects**: The root `ontology.ttl` has syntax errors that block standard RDF parsers.
