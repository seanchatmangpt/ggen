# Handoff Report: Detailed Audit of `crates/ggen-marketplace` Rust Core Crate

## 1. Observation

Detailed code inspections of `crates/ggen-marketplace` identified the following concrete implementation details:

### A. Cache Verification Failure (`src/marketplace/cache.rs` and `src/marketplace/install.rs`)
- In `src/marketplace/cache.rs` (lines 458-494), `verify_digest` walks the cache directory to compute a SHA-256 digest:
  ```rust
  pub fn verify_digest(&self, pack: &CachedPack) -> Result<bool> {
      ...
      // Walk the pack directory and hash all files
      if pack.cache_path.exists() {
          for entry in walkdir::WalkDir::new(&pack.cache_path)
              .into_iter()
              .filter_map(|e| e.ok())
          {
              if entry.file_type().is_file() {
                  if let Ok(contents) = fs::read(entry.path()) {
                      hasher.update(&contents);
                  } else {
                      verified = false;
                      break;
                  }
              }
          }
      }
      let calculated_digest = hex::encode(hasher.finalize());
      let matches = calculated_digest == pack.digest;
      ...
  ```
- In `src/marketplace/install.rs` (line 447), `digest` is calculated over the raw, compressed `pack_data` archive:
  ```rust
  // Calculate final digest
  let digest = ChecksumCalculator::calculate(&pack_data);
  ```
- Because `WalkDir` does not guarantee any deterministic traversal order, the directory hash is non-deterministic.
- Additionally, `pack.digest` holds the hash of the compressed archive (`zip` or `tar.gz` bytes), which will never match the hash of the individual uncompressed files inside the cache directory.

### B. Non-Deterministic Receipt Hash (`src/marketplace/composition_receipt.rs`)
- In `src/marketplace/composition_receipt.rs` (lines 179, 203), the maps are defined as standard `HashMap`s:
  ```rust
  pub versions: HashMap<String, String>,
  ...
  pub ownership_map: HashMap<String, OwnershipRecord>,
  ```
- In `compute_receipt_id` (lines 487-498), the ID is generated by serializing the receipt to JSON:
  ```rust
  pub fn compute_receipt_id(&mut self) -> Result<()> {
      let _ = self.receipt_id.take();
      let json = self.to_json()?;
      let hash = sha2_digest(&json);
      self.receipt_id = Some(hash.clone());
      Ok(())
  }
  ```
- Because `HashMap` iteration order is non-deterministic and varies due to randomized hashing, serializing a receipt with identical data produces different JSON outputs and therefore different cryptographic hashes across runs.

### C. Insecure Archive Extraction & HTTP Connection Pooling (`src/marketplace/install.rs`)
- In `src/marketplace/install.rs` (line 487), `download_pack` instantiates a new HTTP client on every call:
  ```rust
  let client = Client::new();
  ```
- In `extract_pack` (lines 766-798), archives are extracted directly into the persistent cache directory `fs::create_dir_all(&cache_path)` using `ZipArchive::extract` and `Archive::unpack` without validating file paths against path traversal (Zip Slip) or performing the extraction in a temporary directory followed by an atomic rename.

### D. RDF Registry Class Hardcoding (`src/marketplace/rdf_mapper.rs`)
- In `src/marketplace/rdf_mapper.rs` (line 633), reconstructed `ReleaseInfo` hardcodes `registry_class`:
  ```rust
  registry_class: crate::marketplace::models::default_registry_class(),
  ```
- `default_registry_class` (in `models.rs` line 561) returns `RegistryClass::Public`. The true registry class (e.g., `PrivateEnterprise`) is not mapped to RDF or recovered from it, making RDF-backed registries identify all packages as public.

### E. Naive SPARQL Injection Prevention (`src/marketplace/rdf/rdf_control.rs`)
- In `src/marketplace/rdf/rdf_control.rs` (lines 391-409), `detect_injection` matches against case-sensitive uppercase strings:
  ```rust
  fn detect_injection(&self, query: &str) -> bool {
      let suspicious_patterns = [
          "DROP",
          "DELETE WHERE {",
          "INSERT DATA {",
          "CLEAR GRAPH",
          "; DELETE",
      ];
      for pattern in &suspicious_patterns {
          if query.contains(pattern) {
              return true;
          }
      }
      false
  }
  ```
- Lowercase keywords (e.g. `"drop"`) bypass this check, while valid text (like description containing `"Dropbox"`) triggers false-positive blocks.

### F. Dead/Unused Code in FMEA Mitigations (`src/marketplace/rdf/fmea_mitigations.rs`)
- In `src/marketplace/rdf/fmea_mitigations.rs`, methods such as `mitigate_malformed_triple`, `mitigate_orphaned_resource`, and `mitigate_query_timeout` are never called by the control plane or repository. They are entirely dead code only compiled for tests.

### G. Unimplemented Custom Policy Rules (`src/marketplace/policy.rs`)
- In `src/marketplace/policy.rs` (lines 513-518), custom policy rules cause immediate failure:
  ```rust
  PolicyRule::CustomSparql { .. } | PolicyRule::CustomShell { .. } => {
      return Err(Error::ValidationFailed {
          reason: "Custom policy rules require execution context".to_string(),
      });
  }
  ```
- If `CustomShell` were implemented, it would execute arbitrary shell commands from policy packs, posing a severe RCE risk.

### H. Typestate Trust Tier Logic Issues (`src/marketplace/trust.rs`)
- In `src/marketplace/trust.rs` (lines 53-71), trust priority is defined such that `Blocked` is 7. As a result, `meets_requirement` allows non-blocked tiers to satisfy a `Blocked` requirement, and quarantined packages (priority 5) satisfy experimental (priority 6) minimum requirements.

## 2. Logic Chain

1. **Cache Verification Failure**:
   - `WalkDir` yields directory entries in filesystem-dependent order.
   - Hashing files in a variable order results in a variable digest.
   - Comparing this variable digest against the static archive hash (`pack.digest`) guarantees a mismatch.
   - *Conclusion*: Cache hits always fail verification, causing the system to purge the cache and re-download packages on every installation.

2. **Non-Deterministic Receipt Hash**:
   - Iteration order of `HashMap` is randomized by Rust's hasher.
   - `serde_json::to_string_pretty` serializes map fields in iteration order.
   - *Conclusion*: Identical receipts produce different JSON strings and SHA-256 hashes, preventing consistent chain of evidence verification.

3. **Insecure Archive Extraction & HTTP Pooling**:
   - Re-instantiating `Client` prevents HTTP connection reuse, slowing down download batches.
   - Extracting directly to the final cache path without checking filenames for `..` allows a malicious archive to overwrite files outside the cache directory (Zip Slip).
   - If extraction is interrupted, a partially extracted pack remains in the cache, corrupting subsequent builds.

4. **RDF Registry Class Hardcoding**:
   - Reconstructing packages from RDF forces the registry class to `RegistryClass::Public`.
   - Security profiles (like `regulated_finance_profile`) forbid public registry packages.
   - *Conclusion*: Private packages loaded via RDF are blocked by enterprise policies because their private registry status was discarded during RDF round-tripping.

5. **Naive SPARQL Injection Prevention**:
   - Case-sensitive matching of `"DROP"` is easily bypassed via lowercase `"drop"`.
   - Normal text containing substrings like `"DROP"` is incorrectly blocked.

6. **Unimplemented/Unsafe Custom Policy Rules**:
   - Policies containing custom rules cannot be enforced and fail immediately.
   - If shell execution were enabled, malicious policy packs could execute arbitrary commands on the build machine.

---

## 3. Caveats

- We assumed that `Cargo.toml` did not configure `serde_json` with the `preserve_order` feature. We confirmed this by viewing `Cargo.toml`, which showed `serde_json = "1.0"` with no features enabled.
- We did not audit the `oxigraph` store implementation internals, assuming its query engine conforms to standard SPARQL 1.1 specs.
- The unit tests could not be run to completion locally due to an active build directory lock held by another process.

---

## 4. Conclusion

The `crates/ggen-marketplace` crate contains multiple correctness and security issues:
1. **Critical Perf Bug**: Pack caching is entirely broken because `verify_digest` compares a non-deterministic directory walk hash to a compressed archive hash.
2. **Cryptographic Defect**: Receipt IDs are unstable due to `HashMap` serialization ordering.
3. **Enterprise Bypass**: Private registry metadata is lost when stored in RDF, causing false positives in regulated environments.
4. **Security Risks**: Naive SQL/SPARQL injection checks are bypassable, and zip/tar extraction paths are vulnerable to traversal and corruption.
