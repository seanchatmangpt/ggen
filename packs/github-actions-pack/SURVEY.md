## Aggregate observations for gate design

1. Zero workflows use `write-all`. Two lack a top-level `permissions:` block entirely
   (`generate-release-notes.yml`, `marketplace-docs.yml` — both compensate with job-level
   blocks; a gate on "unspecified permissions" must accept job-level as satisfying).
2. Zero `pull_request_target` triggers anywhere — good baseline.
3. `automated-rollback.yml` pipes `secrets.KUBECONFIG_PROD` through `echo` into a file —
   secret-use-without-declared-reason candidate for the gates layer.
4. Concurrency groups are near-universal (23/24; tf-acceptance.yml lacks one).
5. `timeout-minutes` is near-universal at job level; tf-acceptance.yml's job lacks it.
6. Matrix use: erlang-ci (otp), erlang-release (otp x os), release.yml (os x2 + repro
   verify). ci.yml deliberately matrix-free with documented rationale.
7. Runner set: ubuntu-latest everywhere except `runs-on: ${{ matrix.os }}` in
   erlang-release.yml and release.yml.
