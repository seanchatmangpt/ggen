# Vision 2030 Self-Audit and Coverage Summary

This document provides a high-level summary of the process events, object types, event types, relationship qualifiers, and compliance verification status recorded during the development of the `ggen-graph` crate, as captured in `crates/ggen-graph/audit/vision2030.self_audit.ocel.json` and `crates/ggen-graph/audit/vision2030.coverage.json`.

---

## 1. High-Level Summary of Process Events and Objects

The development and validation lifecycle of `ggen-graph` is modeled as an Object-Centric Event Log (OCEL) under the Vision 2030 quality framework. This framework ensures that code artifacts, test suites, compliance checks, and cryptographic receipts are causally linked in a verifiable timeline.

- **Process Log (`vision2030.self_audit.ocel.json`)**: Captures **25 distinct process objects** and **17 process events** chronologically. The event sequence traces the path from initial requirement declaration, through ontology mapping and implementation changes, up to anti-fake/forbidden boundary compliance scans, test runs, cryptographic receipt verification, and the final promotion evaluation of the `self_audit_verification` checkpoint.
- **Coverage Matrix (`vision2030.coverage.json`)**: Contains **9 distinct requirements** (designated from `req_r1_one_crate` to `req_r9_proof_report`), mapping each to its respective implementation source files, test files, and validation command boundaries.

## 2. Process Ontology

### 18 Object Types
The self-audit model utilizes exactly **18 Object Types** to categorize all process actors, resources, and decisions:

1. **`RustCrate`**: Represents the compilation package boundary (`ggen-graph`, version `0.1.0`).
2. **`PRDRequirement`**: A Product Requirement Document constraint (e.g., R1, R2, R5, R6, R7, R8, R9).
3. **`ARDRequirement`**: An Architectural Requirement Document constraint (e.g., R3, R4).
4. **`GALLCheckpoint`**: A checkpoint gate that controls milestone progress (`obj_cp_verify`).
5. **`PublicOntology`**: The prefix and namespace definition of the public schema (`ocel` at `http://www.ocel-standard.org/ns#`).
6. **`OntologyTerm`**: A semantic term definition within the ontology schema (e.g., `Event`).
7. **`SourceFile`**: An implementation file containing functional production code.
8. **`TestFile`**: A test suite verifying functional correctness.
9. **`ExampleFile`**: A demo script showing usage.
10. **`FixtureFile`**: Pre-materialized reference data used in verification tests.
11. **`ScriptFile`**: A validation script executing policy rules.
12. **`Command`**: An abstract definition of a verification tool command.
13. **`CommandRun`**: A concrete execution instance of a verification command (exit status, host).
14. **`EvidenceArtifact`**: Compiled output or log proof generated by a verification boundary.
15. **`GraphReceipt`**: A cryptographically signed block receipt of a graph transaction.
16. **`CoverageMatrix`**: An object recording evaluated code coverage metrics (e.g., line and branch coverage values).
17. **`PromotionDecision`**: The final status decision representing approval/refusal of the checkpoint.
18. **`UnsupportedCapability`**: An out-of-scope capability exemption declared in the audit trace.

### 17 Event Types
The process history is structured using exactly **17 Event Types (Activities)**:

1. **`RequirementDeclared`**: The initial declaration of PRD and ARD requirements.
2. **`OntologyMapped`**: Mappings between public ontologies and internal vocabulary terms.
3. **`FileEmitted`**: The generation or emission of code/audit files.
4. **`ImplementationChanged`**: Updates to production files satisfying specific requirements.
5. **`FixtureCreated`**: Serialization of a test fixture file.
6. **`CommandExecuted`**: Running of build or verification tool commands.
7. **`TestPassed`**: Successful execution of a Rust test suite.
8. **`TestFailed`**: Unsuccessful test execution (capturing process failure capability).
9. **`ForbiddenSurfaceScanned`**: Scans checking for forbidden code surfaces (e.g., mocks/stubs/fake clients).
10. **`AntiFakeScanned`**: Audits checking for placeholder, mock, or fake-success patterns.
11. **`ReceiptEmitted`**: Emission of a cryptographically signed block/transaction receipt.
12. **`ReplayVerified`**: Replaying and verifying cryptographic receipts.
13. **`CoverageEvaluated`**: Computation of coverage matrix statistics.
14. **`CheckpointEvaluated`**: Verification of checkpoint criteria using commands and runs.
15. **`CheckpointPromoted`**: Official approval and promotion of a checkpoint.
16. **`CheckpointRefused`**: Official rejection of a checkpoint due to validation failure.
17. **`UnsupportedCapabilityDeclared`**: Logging of out-of-scope capabilities, linked to source requirements.

## 3. Relationship Qualifiers

The connections between events and objects are qualified using five explicit qualifiers, ensuring semantic precision in process queries:

- **`--checks-->`**: Applied when an event scans, maps, or runs verification checks on a policy, checkpoint, test, or ontology term (e.g., a script checking a source file, or an evaluation checking a checkpoint).
- **`--produces-->`**: Applied when an event generates a physical file, executes a command run, or issues a cryptographic receipt (e.g., file emission producing a source file, or an execution producing a command run).
- **`--verifies-->`**: Applied when a test or replay event validates the correctness of a code file, receipt, or artifact.
- **`--satisfied_by-->`**: Applied when a requirement declaration or checkpoint evaluation is mapped to the implementing resource or confirming command execution.
- **`--decides-->`**: Applied when a decision event registers a formal checkpoint promotion/refusal or unsupported capability declaration.
