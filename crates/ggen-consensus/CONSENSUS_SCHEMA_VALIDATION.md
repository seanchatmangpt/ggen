# Byzantine Fault Tolerant Consensus with ggen-Generated Schemas

## Overview

This document describes the integration test that validates Byzantine Fault Tolerant (PBFT) consensus using schemas generated by `ggen` (the RDF-based code generation tool).

**Test Location**: `crates/ggen-consensus/tests/schema_consensus_validation.rs`

## Concept

The test demonstrates a critical property of distributed systems: **when multiple nodes use identical validation rules (generated from the same schema), Byzantine Fault Tolerant consensus can ensure agreement even when some nodes are dishonest**.

### Why This Matters

1. **Generated schemas ensure consistency**: All nodes load schemas from `ggen` output, guaranteeing identical validation logic
2. **Consensus provides Byzantine safety**: Even with malicious nodes voting dishonestly, honest nodes reach agreement
3. **Schema + Consensus = Trust**: Data validation + BFT consensus = trustworthy distributed agreement

### Generated Schema Example

From `examples/openapi-variants/golden/schemas.js`:

```javascript
export const userSchema = z.object({
  bio: z.string().max(500).optional().nullable(),
  email: z.string().email("Must be a valid email address"),
  id: z.string().min(1),
  posts: z.lazy(() => z.array(postSchema)).default([]).optional().nullable(),
  username: z.string().min(1).max(255),
});
```

### Test Schema Replica (Rust)

The test implements these same validation rules in Rust:

```rust
pub fn validate_user(data: &str) -> ValidationResult {
    // Validate required fields (min(1) constraint from ggen schema)
    if u.id.is_empty() || u.username.is_empty() || u.email.is_empty() {
        return ValidationResult::Invalid;
    }
    // Validate email format
    if !u.email.contains('@') {
        return ValidationResult::Invalid;
    }
    // Validate username length (max(255) constraint from ggen schema)
    if u.username.len() > 255 {
        return ValidationResult::Invalid;
    }
    // Validate bio length (max(500) constraint from ggen schema)
    if let Some(ref bio) = u.bio {
        if bio.len() > 500 {
            return ValidationResult::Invalid;
        }
    }
    ValidationResult::Valid
}
```

**Objective**: Even when 1 out of 4 nodes votes dishonestly, honest nodes reach correct consensus

**Test**: `test_schema_consensus_byzantine_tolerance_4_nodes_1_fault`

- Max faults (f): 1
- Consensus threshold: 2f + 1 = 3 votes

## PBFT Consensus Quorum Requirements

### Formula
For n = 3f + 1 replicas:
- **Consensus threshold**: 2f + 1 votes
- **Byzantine tolerance**: f malicious nodes

## Key Properties Demonstrated

1. **Schema Consistency**: All nodes validate data identically
2. **Honest Consensus**: When nodes are honest, consensus matches schema validation
3. **Byzantine Safety**: Dishonest nodes cannot change the consensus decision (2f+1 threshold prevents this)
4. **Quorum Calculation**: PBFT quorum math ensures consensus is always reachable with 3f+1 nodes
5. **Deterministic Validation**: Same data + same schema = same result across all nodes

## Integration with ggen

### How to generate real schemas

1. Create RDF specification:
   ```bash
   mkdir -p .specify/specs/user-schema
   vim .specify/specs/user-schema/schema.ttl
   ```

2. Generate Zod schemas:
   ```bash
   ggen sync --audit true
   ```

3. The generated schemas would be placed in `lib/schemas/` and can be imported into consensus nodes

### Current Test Status

The test currently uses **hand-crafted schemas** that replicate ggen-generated patterns:
- Matches actual Zod schema constraints from `examples/openapi-variants/golden/schemas.js`
- Demonstrates how real ggen-generated schemas would be used
- Can be extended to load actual generated schemas when available
