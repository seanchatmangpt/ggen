KEY DESIGN DECISIONS
================================================================================

REPLICATION STRATEGY
  • Vector clock causal ordering (not wall-clock timestamps)
  • Rationale: Clock skew immune, deterministic across regions
  • Regions: US-East (primary), US-West (replica), EU (replica)

CONSISTENCY MODEL
  • Phase 1: Eventual consistency with causal ordering
  • Phase 2: CRDT-based merge (zero data loss)
  • Phase 3: Raft consensus (linearizability)

CONFLICT RESOLUTION
  • Phase 1 (Q2 2026): Last-Write-Wins
    - Deterministic winner (VC ordering + region ID tiebreaker)
    - Fast (10ms) but accepts ~5% data loss
    - Rationale: Ship fast, iterate with feedback
  
  • Phase 2 (Q3 2026): CRDT Merge
    - LWW-Element-Set for goals/domains (union)
    - OR-Set for hierarchical adds/removes
    - G-Counter for metrics
    - Zero data loss, slightly slower (50ms)
  
  • Phase 3 (Q4 2026): Application-Level
    - Custom merge logic for workflows, policies
    - Confidence scoring (auto-apply vs manual review)
    - Rationale: Domain experts design optimal merge

RECOVERY STRATEGY
  • 5-stage deterministic process (300 seconds total)
    Stage 1: Vector clock sync (30s)
    Stage 2: State snapshot (60s)
    Stage 3: Event replay (60s)
    Stage 4: Verification (30s)
    Stage 5: Handoff (10s)
  
  • Abort conditions: Checksum mismatch, VC divergence, timeout
  • Rationale: Faster than consensus-based (5min vs 10+ min)

EVIDENCE LEDGER
  • Immutable, hash-chained receipts across 3 regions
  • Storage: Firestore (region-local) + Cloud Logging (mirror)
  • Archive: GCS (7-year retention, tamper-proof)
  • Rationale: Compliance + auditability, separates proof from mutable data

FAILURE SCENARIOS COVERED
================================================================================

1. SINGLE REGION FAILURE (US-West down)
   Impact: 30% throughput loss, 50ms latency increase
   Recovery: Automatic rejoin via 5-stage process (5 minutes)
   Evidence: Recovery receipt captures events replayed

2. NETWORK PARTITION (split-brain)
   Scenario: US-East/West connected, EU isolated
   Handling: Majority partition (2 regions) serves reads/writes
            Minority partition (1 region) enters read-only mode
            LWW tiebreaker resolves conflicts on heal
   Evidence: Conflict record captured, operator alerted

3. CASCADING FAILURES (all replication channels failing)
   Handling: Graceful degradation to local-only mode
            Writes queue locally, replication resumed on recovery
   Evidence: Degradation metrics track mode transitions

4. SPLIT-BRAIN (both partitions commit writes)
   Example: Policy-1 updated two ways during partition
            East: v2 (reduce-cost goal)
            West: v2' (improve-safety goal)
   Resolution: VC tiebreaker (east < west lexicographically)
              East version wins, loser lost
   Evidence: Conflict log + operator notification

VECTOR CLOCK EXAMPLE
================================================================================

Scenario: Three regions, two concurrent updates, conflict resolution

Initial State:
  All regions: policy-1 = "v1", VC = [0, 0, 0]

US-East Timeline:
  T0: policy-1 update → "v2" (reduce-cost)
      VC = [1, 0, 0]
  T1: Receives EU update (VC = [0, 0, 1])
      VC merge = [1, 0, 1]

US-West Timeline:
  T0: policy-1 update → "v2'" (improve-safety)
      VC = [0, 1, 0]
  T1: Receives both updates
      VC merge = [1, 1, 1]

EU Timeline:
  T0: policy-1 update → "v2''" (different)
      VC = [0, 0, 1]
  T1: Receives East update (VC = [1, 0, 0])
      VC merge = [1, 0, 1]
  T2: Receives West update (VC = [0, 1, 0])
      VC merge = [1, 1, 1]

Conflict Detection:
  East: v2, VC = [1, 0, 0]
  West: v2', VC = [0, 1, 0]
  Neither dominates! → CONFLICT

Conflict Resolution (LWW):
  Tiebreaker: "east" < "west" lexicographically
  Winner: East (v2, reduce-cost)
  Loser: West (v2', improve-safety) ← LOST

Result: All regions converge to "v2" after replication completes

KEY INNOVATIONS
================================================================================

1. VECTOR CLOCK CAUSAL ORDERING
   • Replaces wall-clock timestamps (immune to clock skew)
   • Deterministic conflict resolution (same outcome everywhere)
   • Enables true eventual consistency (no ambiguity)

2. PHASE-BASED ROLLOUT
   • Phase 1 (LWW): Fast delivery in 3 months
   • Phase 2 (CRDT): Zero data loss for commutative ops
   • Phase 3 (Consensus): Strongest consistency available
   • Allows iterative improvement based on feedback

3. EVIDENCE LEDGER SEPARATION
   • Immutable ledger separate from mutable data
   • Hash-chain proves tamper-detection
   • Complies with audit + compliance requirements
   • Cryptographic proof of all actions

4. GRACEFUL DEGRADATION
   • Automatic fallback from replicated → local-only mode
   • Prevents cascading failures
   • User-facing latency stays reasonable
   • Transparent recovery when connectivity restored

5. DETERMINISTIC RECOVERY
   • 5-stage process with verification at each step
   • Faster than consensus-based (5min vs 10+ min)
   • Manually auditable (operator can verify each stage)
   • Abort on verification failure (fail-safe)

================================================================================
RISK MITIGATIONS
================================================================================

Risk: Data Loss During Conflicts (Phase 1)
  Mitigation 1: Conflict log + operator notification
  Mitigation 2: Phase 2 CRDTs for commutative operations
  Mitigation 3: Evidence ledger captures losing version

Risk: Clock Skew (if using wall-clock timestamps)
  Mitigation: Vector clocks (immune to skew)

Risk: Cascading Failures
  Mitigation 1: Graceful degradation to local-only mode
  Mitigation 2: Circuit breaker stops failed replication
  Mitigation 3: Bounded recovery time (5 minutes max)

Risk: Split-Brain Consistency
  Mitigation 1: Majority partition rule (2/3 regions)
  Mitigation 2: Minority partition enters read-only mode
  Mitigation 3: LWW deterministic resolution on heal

Risk: Long Recovery Times
  Mitigation 1: 5-stage process with parallelization (5min total)
  Mitigation 2: State snapshot + incremental event replay
  Mitigation 3: Abort on failure (operator intervention)

Risk: Evidence Ledger Tamper
  Mitigation 1: Hash-chain proves modifications
  Mitigation 2: Cloud Logging mirror (tamper-proof)
  Mitigation 3: GCS archival (7-year retention, immutable)
