# Receipt-Dense Combinatorial Maximalism

## A PhD Thesis on ggen, OCEL, AutoML, and Ontology-Driven Proof Manufacture

### Thesis Claim

Combinatorial Maximalism becomes economically viable only when the system maximizes the information density of receipts while minimizing the labor required to produce them. ggen is the missing manufacturing layer because it converts ontology, process evidence, and artifact traces into receipt structures automatically. OCEL supplies the multi-object event substrate, AutoML supplies the self-configuring evidence-design principle, and ggen supplies the deterministic μ-machine that manufactures receipts from public meaning-bearing structure rather than requiring humans or agents to hand-author proof.

---

## Abstract

Combinatorial Maximalism demands that every important claim be proven across its possibility space: successful behavior, correct refusal, invariant preservation, real boundary traversal, visible evidence, and release binding. At first glance, this seems expensive. If every algorithm, route, app feature, package release, and care process must emit high-quality receipts, then the receipt burden appears larger than the implementation burden itself.

This thesis argues the opposite: Combinatorial Maximalism becomes practical precisely when receipt structure is not manually written. The ontology must manufacture the receipt. ggen becomes the first consistent system because it turns public graph-aligned specification, process evidence, and deterministic transformation traces into high-density proof artifacts with low human and agent effort.

The key equation is:

A + μ traces → ontology-aligned event/object evidence → receipt structure → O*

The system starts from artifacts and manufacturing traces, not from an already-known closure. It observes what happened, identifies the objects involved, binds activities to evidence, emits receipts, and reconstructs the lawful source structure. OCEL is essential because real work is not single-case. A release involves algorithms, examples, scripts, packages, tarballs, commits, receipts, failure codes, and verifier runs. A care route involves people, requests, messages, consent, owners, follow-up, and closure. OCEL-like thinking captures the multi-object nature of work instead of forcing it into a single linear case.

AutoML principles supply the second missing idea: the human should not have to design every proof by hand. In AutoML, the system searches, configures, evaluates, and selects model pipelines. In receipt-first Combinatorial Maximalism, the ontology plays the same role for proof: it determines what evidence must exist, what cases must run, what failures must be refused, what receipts must be emitted, and what shape those receipts must take. ML-Schema is relevant here because it was designed to represent and interchange information about ML algorithms, datasets, and experiments across systems, making it a public anchor for describing algorithm evidence rather than inventing private proof classes. ([arXiv][1])

The conclusion is that Combinatorial Maximalism is not a demand for humans or AI agents to do more clerical work. It is a demand to move proof generation into the manufacturing substrate. ggen is therefore not merely a code generator. It is a receipt-density engine.

# 1. The Problem: Proof Is Usually Too Expensive

The traditional failure mode of AI-assisted development is not only incorrect code. It is cheap claims.

An agent says:

It works.

The tests pass.

The package is ready.

The release is closed.

The algorithm is verified.

The care route is complete.

The system has no gap.

These claims are cheap because language is cheap. Proof is expensive.

Combinatorial Maximalism reverses the cost structure. It says that the claim must carry enough information to reconstruct the route by which it became true. A useful receipt must answer:

What object moved?

Which activity moved it?

Which agent or script performed the action?

Which input was used?

Which output was produced?

Which boundary was crossed?

Which failure mode was refused?

Which hash binds the result?

Which version and commit does it belong to?

Which next state is now admissible?

That is a large information payload. If humans or coding agents must hand-author it, the system will collapse under proof overhead.

Therefore the core design problem is:

How do we maximize receipt information density while minimizing receipt authoring effort?

The answer is ontology-driven receipt manufacture.

---

# 2. Information Density as the Core Metric

A receipt is not useful because it exists. A receipt is useful because it compresses the right evidence into a verifiable object.

A low-density receipt says:

Algorithm passed.

A high-density receipt says:

Algorithm dfg was present in the registry, reachable through the dispatcher, available through the CLI, bound to the correct package version, run against a specific input hash, produced a specific output hash, completed in a measured duration, emitted a receipt hash, and refused an empty log with the structured failure code EMPTY_EVENT_LOG without panic, fallback, or false success.

Both are short. Only the second is dense.

Receipt density can be defined as the amount of reconstructive power per unit of human effort.

High-density receipt:

Allows reconstruction of activity, object, agent, input, output, boundary, failure mode, and validity.

Low-density receipt:

Only repeats the agent’s claim.

The aim of ggen is to make high-density receipts cheap.

The human should not write 60 algorithm behavior receipts.

This is the critical distinction:

Combinatorial Maximalism is maximalist in evidence coverage, not maximalist in human effort.

That is where ggen enters.

# 5. OCEL as the Receipt Substrate

OCEL is essential because receipt-rich systems are multi-object systems.

Traditional event logs often assume a single case identifier. That works for narrow workflow analysis but breaks in real systems. Object-centric event logs were introduced precisely because real processes involve multiple objects, and OCEL can capture events connected to multiple object types. Research on OCEL emphasizes that real-life processes often involve multiple objects and that object-centric representations support analysis from different angles rather than forcing one case notion. ([arXiv][2])

That maps exactly to receipts.

A release event is not about one object. It involves:

Fixture.

Invariant case.

Evidence file.

Receipt.

Release certificate.

NPM registry artifact.

A prayer route is not about one object. It involves:

Prayer request.

Consent.

Route.

Message.

Care owner.

Follow-up.

Receipt.

Admin surface.

Visibility policy.

A generated artifact is not about one object. It involves:

Therefore, the receipt system must be object-centric. It must not ask, “What is the case ID?” It must ask, “Which objects participated in this event, and what did the event prove about each one?”

This is why OCEL is not an optional analytics format. It is the natural evidence substrate for receipt-dense Combinatorial Maximalism.

# 9. Public Ontology Alignment Without Private Namespace

The receipt system must avoid private vocabulary lock-in.

The point is not to invent a private proof language. The point is to align receipts to public vocabularies.

Use:

PROV-O for entities, activities, agents, usage, generation, attribution, derivation.

DCAT for evidence datasets and distributions.

ML-Schema for algorithms, runs, datasets, experiments.

SKOS for failure codes, categories, deployment profiles.

SHACL for validation constraints.

SPDX checksum terms for hashes.

schema.org and DOAP for software package identity.

ODRL for policy/refusal constraints.

OWL-Time for timestamps and intervals.

PROV is particularly important because it is designed to describe the entities, activities, and agents involved in producing data or things so that people can assess quality, reliability, and trustworthiness. ([Wikipedia][3])

This means the receipt is not a private artifact. It is a public evidence object.

The system can emit JSON for convenience, but the structure should project cleanly into public RDF alignment.

That is how O* becomes publicly grounded instead of privately asserted.

# 11. Receipt Information Theory

A receipt has value when it reduces uncertainty.

The uncertainty is:

Did this happen?

Did it happen through the real boundary?

Did it use the right input?

Did it produce the right output?

Did it fail correctly when invalid?

Did it bind to the current artifact?

Did it belong to the release?

Can it be replayed?

Can it be independently verified?

A low-density receipt reduces little uncertainty.

A high-density receipt reduces much uncertainty.

The ideal receipt has high mutual information with the underlying route.

That means:

Given the receipt, one can reconstruct the route.

Given the route, one can predict the receipt.

This is the closure pair.

In formula form:

Receipt density is high when R preserves the maximum reconstructive information about A, μ, objects, boundaries, and outcomes while requiring minimal human or agent input.

The system should optimize for:

Maximum reconstructive information.

Minimum authoring burden.

Maximum verifier reproducibility.

Minimum narrative trust.

This is the information-theoretic heart of ggen for Combinatorial Maximalism.

# 14. The ggen Receipt Pipeline

The mature ggen receipt pipeline should look like this:

Input layer:

Public-aligned ontology.

Artifact inventory.

μ trace inventory.

Fixture inventory.

Failure code concept scheme.

Process-object model.

Extraction layer:

SPARQL queries identify algorithms, routes, artifacts, cases, commands, packages, and evidence obligations.

Construction layer:

CONSTRUCT queries form the receipt graph.

Generation layer:

Templates render JSON evidence, Markdown matrices, SHACL shapes, verifier scripts, package certificates, and app-visible receipt surfaces.

Canonicalization layer:

Outputs are sorted, normalized, hashed, and made deterministic.

Verification layer:

Receipts are recomputed, evidence files checked, hashes bound, and missing obligations refused.

Projection layer:

The same evidence can render as:

Release certificate.

Algorithm behavior matrix.

Admin receipt view.

Package proof.

Public RDF graph.

Human-readable paper appendix.

This is ggen as receipt operating system.
